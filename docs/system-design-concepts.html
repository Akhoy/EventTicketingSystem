1. CAP Theorem

Any distributed system can only guarantee two of three properties simultaneously:

LetterPropertyPlain English
CConsistencyEvery read returns the most recent write. No stale data.
AAvailabilityEvery request gets a response. System never refuses.
PPartition ToleranceSystem keeps working even when nodes can't talk to each other.
P is not optional. Networks fail in every real distributed system. So the real choice is always CP or AP — never CA.

CP — Consistency + Partition Tolerance

When a partition occurs, the system rejects requests rather than risk returning stale or inconsistent data. Availability is sacrificed.

Your example: Redis seat counter. If Redis is unreachable, /book returns an error. Better to be down than to oversell a seat.

AP — Availability + Partition Tolerance

When a partition occurs, the system keeps responding — possibly with stale data. Consistency is sacrificed.

Your example: MongoDB AvailableSeats. CatalogSyncWorker may be behind. GET /items still responds with last known values. A slightly wrong seat count on the browse page is acceptable.

Your system's CAP trade-offs

ComponentChoiceWhy
Redis (seat counter)CPOverselling is unrecoverable. Downtime is recoverable.
MongoDB (catalogue)APStale seat count on browse page is acceptable. Unavailability is not.
SQL Server (bookings)CPACID. Source of financial truth.
RabbitMQ + OutboxAPMessages delayed if broker down, never lost. Eventual consistency.
Interview answer: "I made different CAP trade-offs per data store based on the cost of being wrong. Redis and SQL are CP — overselling a seat is unrecoverable so consistency wins. MongoDB is AP — a slightly stale seat count on a browse page is a business-acceptable trade-off. Availability wins there."

2. ACID

ACID is a set of four guarantees a database makes about transactions. A transaction is a group of operations that must all succeed or all fail together.

A — Atomicity

All or nothing. Every operation in a transaction either fully commits or fully rolls back. No half-written state ever exists.

Your example:

using var transaction = await db.Database.BeginTransactionAsync();
booking.Confirm();
await db.SaveChangesAsync();
await transaction.CommitAsync(); // if this fails, nothing was written

If SaveChangesAsync throws, the booking stays Pending. You never get a half-confirmed booking.

Real world: Bank transfer — debit Account A, credit Account B. If the server crashes between them, atomicity guarantees the debit is rolled back. Nobody loses money.

C — Consistency

The database moves from one valid state to another. Rules and constraints always hold.

This C is different from CAP's C. ACID consistency = data follows defined rules (constraints, foreign keys). CAP consistency = every read sees the latest write across distributed nodes. Same letter, completely different meaning.

I — Isolation

Concurrent transactions don't see each other's intermediate state. Controlled by isolation levels (see section 4).

Your example: Two users confirming bookings simultaneously. Transaction 1's half-written state is invisible to Transaction 2.

D — Durability

Once committed, data survives a crash. Written to disk, not just memory.

SQL Server vs Redis: SQL Server writes to a transaction log on disk before confirming a commit. Redis keeps data in memory by default — a restart wipes it. That's why your seat counters need to be repopulated from MongoDB on Redis restart, but your booking records survive SQL Server restarts unchanged.

3. BASE

BASE is the opposite philosophy to ACID. NoSQL databases like MongoDB adopt it.

LetterPropertyPlain English
BBasically AvailableSystem responds most of the time, even if data is stale or partial
SSoft StateState can change over time without a direct write — as sync catches up
EEventually ConsistentGiven enough time, all nodes converge to the same value

Your example: MongoDB AvailableSeats. After a booking is confirmed, CatalogSyncWorker asynchronously decrements the MongoDB count. For 10-30 seconds, MongoDB might show "2000 seats" when Redis says "1995". That's soft state — it's changing without a direct write, as the worker catches up. Eventually both agree. That's BASE.

Interview answer: "My MongoDB catalogue is BASE. AvailableSeats is eventually consistent — it lags behind Redis by however long CatalogSyncWorker takes to process the message. That's acceptable because it's a display value. The booking gate is Redis, which is strongly consistent. I use ACID where money is involved, BASE where approximate data is acceptable."

4. Isolation Levels

Isolation levels control how much concurrent transactions can "see" each other. Higher isolation = more protection, less concurrency, lower throughput.

The Three Problems

Dirty Read

You read data another transaction has written but not yet committed. That transaction might roll back — meaning you read data that never officially existed.

Transaction 1: UPDATE booking SET status = 'Confirmed'  -- not committed yet
Transaction 2: SELECT status FROM booking → 'Confirmed'  -- dirty read
Transaction 1: ROLLBACK                                   -- never happened
-- Transaction 2 acted on data that never existed

Your system impact: OutboxRelayWorker could publish a booking to RabbitMQ, then the HTTP handler rolls back. Catalog.API decrements seats for a booking that never existed.

Non-Repeatable Read

You read the same row twice in one transaction and get different values because another transaction updated it in between.

Transaction 1: SELECT status → 'Pending'
Transaction 2: UPDATE status = 'Confirmed' → COMMIT
Transaction 1: SELECT status → 'Confirmed'  -- different value, same transaction

Same transaction. Same row. Different value. Your logic is now inconsistent with itself.

Phantom Read

You query a set of rows twice in one transaction and get different rows because another transaction inserted or deleted rows in between.

Transaction 1: SELECT COUNT(*) FROM Bookings WHERE EventId = 'evt-coldplay' → 150
Transaction 2: INSERT INTO Bookings (EventId...) VALUES ('evt-coldplay') → COMMIT
Transaction 1: SELECT COUNT(*) FROM Bookings WHERE EventId = 'evt-coldplay' → 151

A new row "appeared" — a phantom. Same query, different result set.

Difference from non-repeatable read: Non-repeatable = same row, value changed. Phantom = same query, different rows returned.

Isolation Levels — What They Prevent

LevelDirty ReadNon-Repeatable ReadPhantom ReadCost
Read Uncommitted❌ Not prevented❌❌Maximum concurrency
Read Committed✅ Prevented❌❌Low — SQL Server default
Repeatable Read✅✅ Prevented❌Medium — locks rows you read
Serializable✅✅✅ PreventedMaximum — kills throughput
Why SQL Server defaults to Read Committed: "It prevents dirty reads — the most dangerous problem — without the throughput cost of Serializable. Serializable locks every transaction so only one runs at a time. Under 50,000 concurrent users that means everything queues behind locks. Read Committed is the pragmatic middle ground."

5. Consistency Models

Between "always right" and "eventually right" there is a spectrum. Different parts of your system sit at different points on it.

ModelMeaningYour example
Strong Consistency Every read returns the latest write. Always. No exceptions. Redis DECR — after it executes, every read of that key sees the decremented value immediately
Read-Your-Writes You always see your own writes immediately. Others might not yet. After confirming a booking, you see "Confirmed" immediately. Another user querying might briefly see "Pending"
Causal Consistency Causally related operations are seen in the correct order by everyone. RabbitMQ — messages within a queue are consumed in order. The cause always appears before the effect.
Eventual Consistency Will converge to the correct value. Timing not guaranteed. MongoDB AvailableSeats — lags behind Redis by seconds. DNS propagation — lags by up to 48 hours.

6. Redis Atomicity & Distributed Counter

What is a Distributed Counter?

A counter that must be accurate across multiple processes running simultaneously. If Booking.API runs as 3 pods, all three pods need to decrement the same count. The counter must live somewhere shared — not in any pod's memory.

Pod 1 → Redis DECR event:evt-coldplay:seats
Pod 2 → Redis DECR event:evt-coldplay:seats  // same key, different pod
Pod 3 → Redis DECR event:evt-coldplay:seats  // all three hit the same Redis key

What is Interleaving?

In a multi-threaded system, the CPU can switch between threads mid-operation. Each "decrement" is actually three steps: read → subtract → write. Interleaving means those steps from different threads mix together:

Thread 1: reads counter = 100
Thread 2: reads counter = 100   // switched before Thread 1 wrote
Thread 1: writes 99
Thread 2: writes 99             // both subtracted 1, counter only moved by 1
// Two bookings happened. One seat consumed. Oversold.

Locks prevent interleaving by forcing threads to take turns. But locks under high concurrency cause contention — threads queue, throughput drops.

Why Redis Doesn't Need Locks

Redis is single-threaded for command execution. Every command runs to completion before the next one starts. No interleaving is possible. No locks needed.

// All three pods fire DECR simultaneously
// Redis processes them one at a time:
Pod 1 DECR → 1999  (complete)
Pod 2 DECR → 1998  (complete)
Pod 3 DECR → 1997  (complete)
// No race condition. No interleaving. No locks.
Interview phrasing: "Redis achieves atomicity through single-threaded command execution, not locking. Every command runs to completion before the next starts. No interleaving is possible so no locks are needed."

Why Single-Threaded Isn't Slow

Redis commands execute in microseconds (millionths of a second). At 50,000 requests per second, that's 50ms of actual work per second. Redis handles over 1 million operations per second on a single thread.

The actual bottleneck is always the network round trip — typically 0.5–2ms. The command itself takes 0.001ms. Single-threaded execution is not the constraint.

Compare to SQL Server: acquire row lock → read → write → release lock → write to transaction log → flush to disk = 5–10ms minimum. Redis: 0.001ms.

The Two Guards in Your System

GuardRedis CommandWhat it prevents
Capacity gate DECR event:{eventId}:seats Total overselling — more bookings than seats exist
Seat lock SET seat:lock:{eventId}:{seatId} NX EX 300 Two users booking the exact same physical seat simultaneously

Why both? The capacity gate alone doesn't prevent two people booking seat A1 — they both pass the capacity check. The seat lock alone doesn't prevent total overselling — it only guards one specific seat. You need both for different failure modes.

Rollback When Counter Goes Negative

long remaining = await cache.StringDecrementAsync(seatsKey);
if (remaining < 0)
{
    await cache.StringIncrementAsync(seatsKey);  // restore counter
    await cache.KeyDeleteAsync(lockKey);          // release seat lock
    return Results.Conflict("Sold out");
}

This is optimistic concurrency with rollback — let the operation happen, check if valid, undo if not. The alternative (pessimistic — check first, then act) requires two separate operations with a race condition between them.

Full interview answer for distributed counter: "I use Redis with two separate guards. First, an atomic DECR on the event capacity counter — Redis is single-threaded so no two DECRs interleave, and if the result goes negative I immediately INCR to restore and reject the request. Second, a SETNX seat lock per specific seat with a 5-minute TTL — this prevents two users booking the exact same seat. If either guard fails I roll back both. SQL is the source of truth for the booking record — Redis is the fast concurrency gate in front of it. The CAP trade-off is CP — if Redis is down we reject requests rather than risk overselling, because an oversold seat is unrecoverable."

7. Domain-Driven Design (DDD)

DDD is a way of writing code that mirrors how the business thinks. The core problem it solves: business rules scattered across the codebase, duplicated, easy to bypass, impossible to test in isolation.

The Core Problem — Anemic Domain Model

An anemic domain model is a class that has data but no behaviour. "Anemic" means weak, lacking vitality. The object just sits there passively while everything external acts on it.

// Anemic — anyone, anywhere can do this
booking.Status = "Confirmed";
booking.Status = "Banana";      // no rules, compiles fine
booking.Status = null;          // crashes later

The business rule "only confirm a Pending booking" exists as an if-statement somewhere in Program.cs. Every developer who touches status must know to check that file. When someone forgets, bugs appear.

The Solution — Aggregate Root

An object that protects its own state. Nobody changes it directly. They call methods.

public class Booking
{
    public string Status { get; private set; } = "Pending";  // private set

    public void Confirm()
    {
        if (Status == "Confirmed") return;           // idempotent
        if (Status != "Pending")
            throw new InvalidOperationException(...); // rule enforced
        Status = "Confirmed";
    }

    public void Expire()
    {
        if (Status != "Pending") return;  // silent no-op — safe to call twice
        Status = "Expired";
    }
}

private set means nothing outside the class can assign to Status. Confirm() is the only door in. The compiler enforces this — trying to do booking.Status = "Confirmed" directly is a compile error on every single build. A code review is a human reading code — a human can miss a line. The compiler cannot. You physically cannot ship the mistake.

The Four Core Concepts

Aggregate Root

An object that controls its own state. The only entry point for changes — no side door. Your Booking class is one.

Plain English: You can't reach SeatId or Status and change them directly — you have to go through a method on Booking. Because all changes go through the same door, all the rules stay intact. There's no way around them.

Entity

An object with an ID. Two entities with the same values but different IDs are different things.

Plain English: Your Booking has a Guid Id. Two bookings for the same seat at the same event are still different bookings because they have different IDs.

Test: If two objects have identical values, are they the same? For an Entity — no, because the ID is different.

Value Object

An object with no ID. Defined purely by its values. Immutable — never modified, only replaced.

Plain English: SeatId("A1") equals another SeatId("A1") because the value is the same. Like a £10 note — you don't care which specific note it is, just that it's £10.

Three benefits:

  • Validation at creation — SeatId("") throws, invalid state is impossible
  • Type safety — compiler catches passing an EventId where SeatId is expected
  • Behaviour on the value — seatId.Row, seatId.IsVIP

Test: If two objects have identical values, are they the same? For a Value Object — yes.

Domain Event

Something that happened. Always past tense. BookingConfirmed. BookingExpired.

Plain English: When Confirm() is called, the booking records that something happened. Other parts of the system react — publish to RabbitMQ, send email, update read model. The booking doesn't know or care what happens next. It just records the fact.

This decouples the domain from infrastructure. The booking's job ends when it raises the event.

Bounded Context

A boundary around a part of the system where words have a specific meaning.

Plain English: "Booking" in Booking.API means a reservation with payment, status, and a 5-minute hold window. "Booking" in Catalog.API means "a seat was sold, decrement the count." Same word, different model, different service.

Why your services are separate: They're in different bounded contexts. Each context owns its model completely. Sharing a database would mean sharing the model — breaking the boundary.

Why BookingEvent DTO is duplicated: Each service owns its own copy of the wire contract. Sharing a type library couples two independently deployable services.

Repository Pattern

Hides database implementation behind an interface. The domain knows nothing about SQL, EF Core, or MongoDB.

public interface IBookingRepository
{
    Task<Booking> GetByIdAsync(Guid id);
    Task SaveAsync(Booking booking);
}
// Domain depends on this interface
// EF Core implementation is hidden behind it

Plain English: The Booking aggregate doesn't call DbContext directly. A repository does the saving. Swap SQL Server for Postgres — change one class. Nothing in the domain changes.

Static Factory Method — Why not just a constructor?

// Static factory method
public static Booking Create(string eventId, string seatId, string userId) { ... }

// Usage
var booking = Booking.Create("evt-coldplay", "A1", "user-123");
  • It has a name — Booking.Create() communicates intent. new Booking() does not.
  • Forces valid state — the only door in. All validation happens here.
  • Private constructor — prevents bypassing Create() via new Booking(). EF Core can still call it via reflection — reflection bypasses access modifiers entirely (private, protected, it doesn't matter). EF Core is the framework, not application code, so it's allowed to do things your own code can't.

Why Confirm() is Idempotent but Expire() Never Throws

The question isn't about the method — it's about who's calling and what stale state means for that caller.

MethodCallerBehaviour on stale stateReason
Confirm() Payment webhook Idempotent (silent) if already Confirmed. Throws if Expired. Webhook may fire more than once — silent return on a duplicate call is the right idempotent behaviour.

But if payment arrives for an Expired booking, something genuinely went wrong — a user paid for a seat they no longer hold. That needs to surface as an error, not be silently swallowed.
Expire() Background sweep worker — runs every 30s across thousands of rows Silent return always — never throws If the worker hits a row that's already Expired or Confirmed, that just means the previous sweep already handled it. Throwing would crash the worker over nothing.

Same situation (stale state), different caller, different right answer.

Interview answer — "Walk me through your Booking class": "Booking is an aggregate root. All properties have private setters — nothing outside the class can change state directly. Business rules live on the object through methods: Confirm() enforces that only a Pending booking can be confirmed and is idempotent so calling it twice is safe — important for payment webhooks that may fire more than once. Expire() is a silent no-op if already expired — the background worker processes many bookings and should never crash on a stale row. If any developer tries to set Status directly, it's a compile error. The rules are enforced by the compiler, not by convention or code review."

8. Clean Architecture

Clean Architecture is a way of structuring a codebase so that business rules never depend on infrastructure. The core rule: inner layers know nothing about outer layers. Dependencies only point inward.

The Problem It Solves

Before the split, everything lived in one project: Booking.cs, BookingDbContext, Program.cs endpoints, and workers all in Booking.API. The problems:

  • Business logic and EF Core calls were in the same files — changing the database means touching the same file as your business rules
  • Workers referenced BookingDbContext directly — persistence and domain logic were tangled
  • No seam to test business rules without a real database

The Three-Layer Split

┌──────────────────────────────────────────────────────┐
│                    Booking.API                       │
│   HTTP endpoints · workers · DI wiring (Program.cs)  │
│   References: Domain + Infrastructure                │
├──────────────────────────────────────────────────────┤
│               Booking.Infrastructure                 │
│   BookingDbContext · BookingRepository · Migrations  │
│   References: Domain only                            │
├──────────────────────────────────────────────────────┤
│                  Booking.Domain                      │
│     Booking aggregate root · IBookingRepository      │
│     Zero NuGet dependencies                          │
└──────────────────────────────────────────────────────┘

Dependency direction:
  API  ──▶  Infrastructure  ──▶  Domain
  API  ──▶  Domain

Domain knows nothing. Infrastructure knows Domain. API knows both.

What Moved Where

FileBeforeAfterWhy
Booking.csBooking.APIBooking.DomainPure business logic. No infrastructure knowledge.
IBookingRepository.csDid not existBooking.DomainDomain defines what it needs, not how it's stored.
BookingDbContext.csBooking.APIBooking.InfrastructureEF Core is an infrastructure detail.
BookingRepository.csDid not existBooking.InfrastructureEF Core implementation of the domain's interface.
Migrations/Booking.APIBooking.InfrastructureMigrations belong with their DbContext.
Why does IBookingRepository live in Domain, not Infrastructure? The domain defines what it needs — a way to save and retrieve bookings. It doesn't define how. The interface is the contract. If it lived in Infrastructure, Domain would have to reference Infrastructure to use it — that reverses the dependency direction. The rule is: Infrastructure depends on Domain, never the reverse.

IServiceCollection vs IServiceProvider

What it isThe registry — list of what can be resolvedThe resolver — hands you an instance at runtime

AddBookingInfrastructure — Extension Method

// Booking.Infrastructure/InfrastructureExtensions.cs
public static IServiceCollection AddBookingInfrastructure(
    this IServiceCollection services, string connectionString)
{
    services.AddDbContext<BookingDbContext>(options =>
        options.UseSqlServer(connectionString));
    services.AddScoped<IBookingRepository, BookingRepository>();
    return services;
}

// Program.cs — one line, no EF Core knowledge here
builder.Services.AddBookingInfrastructure(
    builder.Configuration.GetConnectionString("SqlServer")!);

Program.cs has zero knowledge of BookingDbContext, UseSqlServer, or BookingRepository. All infrastructure wiring is encapsulated in one method in the Infrastructure project. Switching to Postgres means changing UseSqlServer to UseNpgsql in one file — nothing in Program.cs or Domain changes.

Workers are Singleton — they cannot inject Scoped services directly. IBookingRepository and BookingDbContext are Scoped (one instance per request). A Singleton lives forever — holding a Scoped service inside it means that service is never disposed, leaking DB connections. ASP.NET Core throws a captive dependency exception at startup if you try. Solution: inject IServiceScopeFactory (which is Singleton-safe), then create a new scope at the start of each worker cycle. The scope — and the DbContext inside it — is disposed at the end of the using block. 
// Workers do this each cycle:
using var scope = _scopeFactory.CreateScope();
var repository = scope.ServiceProvider.GetRequiredService<IBookingRepository>();
// scope.Dispose() at end of using block → DbContext is cleaned up
Interview answer — "Why Clean Architecture?": "I split Booking.API into three projects: Domain, Infrastructure, and API. Domain has zero NuGet dependencies — pure C# with the Booking aggregate and the IBookingRepository interface. Infrastructure implements that interface using EF Core. API wires everything together with one call to AddBookingInfrastructure(). The dependency direction is always inward: API depends on Infrastructure and Domain, Infrastructure depends on Domain, Domain depends on nothing. If we swap SQL Server for Postgres, nothing in Domain or the HTTP handlers changes — only one method in Infrastructure."

9. Repository Pattern

Why it's called a repository

A repository in the physical world is a place you go to fetch and store things — a library, a warehouse, a filing cabinet. You don't care how things are organised inside. You say "give me the booking with ID 123" and it comes back. You say "store this booking" and it disappears inside. What happens in there is not your problem.

That's exactly what IBookingRepository does. The endpoint says "give me a booking" — it doesn't know or care whether that came from SQL Server, MongoDB, an in-memory list, or a text file. The repository hides all of that.

// The endpoint thinks it's talking to "a place that holds bookings"
var booking = await repository.GetByIdAsync(id);

// It has no idea this is running underneath:
return await _context.Bookings.FindAsync(id);  // SQL Server, EF Core, connection pools — all hidden

The interface is described as collection-like because calling repository.AddAsync(booking) feels like adding to a list — myList.Add(booking). The caller thinks in terms of "a collection of bookings", not "a database table with rows and SQL". The repository creates that illusion: a simple get/add/remove surface, regardless of what's actually inside.

Crucially, the repository holds domain objects — not raw database rows, not DTOs. What comes back from GetByIdAsync() is a Booking domain object, not a plain bag of column values. The distinction matters because a raw row from a DataReader is just data: Id, Status, CreatedAt as primitive values with no behaviour attached. A domain object carries both state and rules. But to be precise — Confirm() and Expire() are not part of the repository. The repository's job is storage, retrieval, and search. You fetch the object from the repository, then call domain behaviour on it, then hand it back to the repository to persist. Three separate steps.

var booking = await repository.GetByIdAsync(id);  // 1. repository retrieves
booking.Confirm();                                 // 2. domain behaviour — nothing to do with repository
await repository.SaveAsync();                      // 3. repository persists the change

Martin Fowler named and defined the pattern in 2002: "a mechanism for encapsulating storage, retrieval, and search behaviour, which emulates a collection of objects." Every developer since has used the same word for the same idea — so when you see IBookingRepository you immediately know what it is without reading the code. The name is convention. The actual value is the interface that lets everything else pretend the database is just a simple collection of domain objects.

The Interface (in Domain)

public interface IBookingRepository
{
    Task AddAsync(Booking booking);
    Task SaveAsync();
    Task<Booking?> GetByIdAsync(Guid id);
    Task<List<Booking>> GetPendingBookingsOlderThan(DateTime cutoff);
    Task<List<Booking>> GetConfirmedUnpublishedBookings();
}

The domain defines what queries it needs. The implementation — EF Core, SQL queries — lives in Infrastructure. Domain never references EF Core.

Before → After: Creating a Booking

// BEFORE — endpoint depended on EF Core directly
app.MapPost("/book/{eventId}/{seatId}", async (BookingDbContext db, ...) =>
{
    var booking = new Booking { EventId = eventId, ... };
    db.Bookings.Add(booking);
    await db.SaveChangesAsync();
});
// AFTER — endpoint depends on the interface only
app.MapPost("/book/{eventId}/{seatId}", async (IBookingRepository repository, ...) =>
{
    var booking = Booking.Domain.Booking.Create(eventId, seatId, userId);
    await repository.AddAsync(booking);
});

Before → After: Confirming a Payment

// BEFORE — raw EF Core + explicit transaction
app.MapPost("/payments/{bookingId}/confirm", async (BookingDbContext db, ...) =>
{
    var booking = await db.Bookings.FindAsync(bookingId);
    using var transaction = await db.Database.BeginTransactionAsync();
    booking.Confirm();
    await db.SaveChangesAsync();
    await transaction.CommitAsync();
});
// AFTER — interface calls, no EF Core, no explicit transaction
app.MapPost("/payments/{bookingId}/confirm", async (IBookingRepository repository, ...) =>
{
    var booking = await repository.GetByIdAsync(bookingId);
    booking.Confirm();
    await repository.SaveAsync();
});

The explicit BeginTransactionAsync / CommitAsync was removed. SaveChangesAsync already wraps its work in an implicit database transaction — for a single-entity save, the explicit transaction was redundant. The implicit transaction guarantees: if SaveChanges fails, the booking stays Pending. No half-confirmed state.

Before → After: BookingExpiryWorker

// BEFORE — worker resolved DbContext, wrote raw LINQ
using var scope = _scopeFactory.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<BookingDbContext>();

var expired = await db.Bookings
    .Where(b => b.Status == "Pending" && b.CreatedAt < cutoff)
    .ToListAsync(stoppingToken);

foreach (var booking in expired)
{
    booking.Expire();
}
if (expired.Count > 0)
    await db.SaveChangesAsync(stoppingToken); // one save after loop
// AFTER — worker resolves the interface, calls named methods
using var scope = _scopeFactory.CreateScope();
var repository = scope.ServiceProvider.GetRequiredService<IBookingRepository>();

var expired = await repository.GetPendingBookingsOlderThan(cutoff);

foreach (var booking in expired)
{
    booking.Expire();
}
// One save after the loop — EF Core tracks all Expire() calls,
// SaveAsync flushes all of them in a single roundtrip.
if (expired.Count > 0)
    await repository.SaveAsync();

The raw LINQ query (db.Bookings.Where(...)) moved into BookingRepository.GetPendingBookingsOlderThan(cutoff). The worker has no idea how bookings are fetched. The single save after the loop is intentional — calling SaveAsync() inside the loop would make N database roundtrips instead of one. EF Core tracks all the Expire() changes in memory; one SaveChangesAsync() flushes them all.

Before → After: OutboxRelayWorker

// BEFORE
foreach (var booking in unpublished)
{
    // publish to RabbitMQ ...
    booking.MarkPublished();
    await db.SaveChangesAsync();
}
// AFTER
foreach (var booking in unpublished)
{
    // publish to RabbitMQ ...
    booking.MarkPublished();
    await repository.SaveAsync(); // still saves per booking — intentional
}

This worker deliberately saves inside the loop, unlike the ExpiryWorker. Reason: if the process crashes mid-loop, already-published bookings have their PublishedAt stamped — they won't be re-published on the next cycle. Crash safety outweighs batching here.

SaveAsync() — Why No Parameter?

// Interface
Task SaveAsync();

// Implementation
public async Task SaveAsync() =>
    await _context.SaveChangesAsync();

SaveChangesAsync() doesn't save a specific booking — it saves all changes tracked by the DbContext. Passing a Booking parameter would imply only that booking is saved, which is misleading. No parameter makes the API honest: "flush everything you're tracking." The caller's pattern is always: call a domain method to change state, then call SaveAsync() to persist those changes.

The "never use repository with EF Core" argument

The short version: "There is literally no point. Just use the DbContext directly. You'll be fine." DbContext is already a repository and Unit of Work. Adding an interface on top is an abstraction over an abstraction.

Here is what the two approaches look like side by side:

// ── WITHOUT a repository — DbContext used directly ───────────────────
app.MapPost("/payments/{bookingId}/confirm", async (
    Guid bookingId,
    BookingDbContext db) =>
{
    var booking = await db.Bookings.FindAsync(bookingId);
    if (booking is null) return Results.NotFound();

    booking.Confirm();
    await db.SaveChangesAsync();
    return Results.Ok();
});

// Need a new query? Write it inline — no interface change needed:
var confirmed = await db.Bookings
    .Where(b => b.Status == "Confirmed")
    .AsNoTracking()
    .Select(b => new { b.Id, b.SeatId })
    .ToListAsync();
// ── WITH a repository ─────────────────────────────────────────────────
// The same new query requires a new method on the interface AND the implementation:

// 1. Add to interface (Domain project):
Task<List<BookingSummaryDto>> GetConfirmedSummaries();

// 2. Implement it (Infrastructure project):
public async Task<List<BookingSummaryDto>> GetConfirmedSummaries() =>
    await _context.Bookings
        .Where(b => b.Status == "Confirmed")
        .AsNoTracking()
        .Select(b => new BookingSummaryDto(b.Id, b.SeatId))
        .ToListAsync();

// Every new query shape = new interface method + new implementation.
// The interface grows indefinitely. With DbContext directly, you just write the query.

The repository implementation can absolutely use IQueryable, AsNoTracking, projections, and all EF Core features internally — the point isn't that those features are unavailable, it's that every distinct query shape still requires a dedicated method on the interface. The interface becomes a growing list of query methods, each defined once in Domain and implemented once in Infrastructure, for what could just be an inline LINQ query at the call site.

Why it's implemented here anyway: Clean Architecture's dependency rule: Domain cannot reference EF Core. IBookingRepository in Domain is the seam that enforces this boundary. Without it, workers and handlers would import Microsoft.EntityFrameworkCore directly. That is the architectural reason — not "we might swap ORMs."
Interview answer — "Walk me through how you abstracted data access": "I introduced IBookingRepository in the Domain layer and implemented it with EF Core in Infrastructure. Before this, Program.cs was calling db.Bookings.Add() and db.SaveChangesAsync() directly — EF Core details leaked into the HTTP handlers. Now endpoints call repository.AddAsync() and repository.SaveAsync() — no EF Core anywhere in Program.cs or the workers. SaveAsync() takes no parameter because SaveChangesAsync saves all tracked changes, not a specific entity. There's a valid argument for just using DbContext directly — it's already a repository and unit of work, and every new query shape means a new interface method. I'd make that argument in a simpler project. Here the repository earns its existence because Domain cannot reference EF Core under the Clean Architecture dependency rule."

10. Unit of Work — and the Nick Chapsas Debate

What is the Unit of Work pattern?

Unit of Work is a pattern that answers one question: when multiple things change in a single business operation, how do you ensure they are all saved together or not at all?

The pattern works by tracking every object that was changed during an operation — created, modified, or deleted — and then flushing all of them to the database in one coordinated transaction when you say "commit". If anything fails, everything rolls back. No partial saves.

Without it, you'd call the database once per change — booking created (1 DB call), status updated (2nd DB call), outbox event written (3rd DB call). If the process crashes between calls 2 and 3, the database is in an inconsistent state. Unit of Work wraps all three into one atomic operation.

How DbContext IS the Unit of Work

EF Core's DbContext implements the Unit of Work pattern natively. You don't need to build one. It works in three phases:

PhaseWhat happensIn code
1. Track Every entity you load or add is registered in the change tracker — an in-memory dictionary of every object the DbContext knows about, along with its original vs current values db.Bookings.Add(booking) or loading via FindAsync()
2. Mutate You change properties on tracked entities in C#. EF Core detects the difference between the original snapshot and the new values automatically — you don't call any "mark as dirty" method booking.Confirm() sets Status = "Confirmed" — EF Core notices
3. Flush One call generates and executes the SQL for every pending change in a single database transaction. All succeed or all roll back await db.SaveChangesAsync() 
// The Unit of Work in action — three changes, one SaveChangesAsync
var booking = await db.Bookings.FindAsync(id);   // tracked by EF Core
booking.Confirm();                               // change tracked (no DB call yet)

var outbox = new OutboxEvent(booking.Id, ...);
db.OutboxEvents.Add(outbox);                     // added to tracker (no DB call yet)

await db.SaveChangesAsync();
// ↑ EF Core now generates two SQL statements inside ONE transaction:
//   UPDATE Bookings SET Status='Confirmed' WHERE Id=...
//   INSERT INTO OutboxEvents (...)
// If either fails, both are rolled back.

This is exactly what repository.SaveAsync() calls under the hood in this project — it delegates to db.SaveChangesAsync(), which is the Unit of Work flush.

The Nick Chapsas debate — full breakdown

Nick Chapsas (a prominent .NET YouTuber) argues that wrapping EF Core in a repository pattern is adding an abstraction over something that is already a well-designed abstraction. His specific points:

ArgumentWhat it means
"DbContext is already a repository" db.Bookings is a DbSet<Booking> — it already provides Add, FindAsync, Where, Remove. IBookingRepository wraps these with methods that do the same thing. You've written an abstraction on top of an abstraction for no new capability.
"DbContext is already a Unit of Work" As shown above — the change tracker + SaveChangesAsync IS the Unit of Work. A separate IUnitOfWork interface wrapping it adds boilerplate without adding behaviour.
"You're never actually swapping the database" The classic justification for the repository is "so you can swap EF Core for Dapper later." In practice this almost never happens — and when it does, the repository interface leaks so many EF Core assumptions (tracked entities, async patterns, IQueryable shapes) that it doesn't actually shield you.
"You lose EF Core's best features" Behind a GetByIdAsync(Guid id) method, callers can't request projections (.Select(b => new Dto(...))), skip tracking (.AsNoTracking()), split queries, or write complex LINQ. You'd have to add a new method to the interface every time a new query shape is needed — the interface grows forever.
"Unit testing argument is weaker than it looks" The repository is often justified as "it lets us mock the database in tests." But EF Core ships an in-memory provider and a SQLite in-memory mode — you can test against a real (lightweight) database without mocking. Mocking the repository hides bugs where your LINQ would fail against a real database.

The counter-arguments — why teams still use it

ReasonWhen it holds
Architectural boundary enforcement In Clean Architecture, Domain must not reference EF Core. IBookingRepository in Domain is the seam — it defines what the domain needs without naming how. Without it, workers and handlers would import Microsoft.EntityFrameworkCore directly, which breaks the dependency rule.
Multiple database backends If the same interface needs SQL and MongoDB implementations (e.g. read from Mongo materialized view, write to SQL), a repository interface is the natural place to express that. DbContext only works with relational databases.
Team consistency On large teams, named methods like GetPendingBookingsOlderThan(cutoff) are more discoverable and harder to misuse than raw LINQ scattered across handlers. The repository becomes a library of vetted query patterns.
Testability via interface For unit testing domain logic in isolation — where you want zero database involvement — an interface you can mock or stub is simpler than configuring an in-memory EF Core provider. This project's BookingExpiryWorker logic could be unit-tested with a fake IBookingRepository that returns hardcoded data.

Why Catalog.API's repository doesn't add a Unit of Work

Catalog.API has a repository (ICatalogRepository) but no separate IUnitOfWork interface. The reason: DbContext is already the Unit of Work. Wrapping it in another interface would mean writing this:

// Pointless wrapper — all it does is delegate to DbContext
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync();
}

public class UnitOfWork : IUnitOfWork
{
    private readonly CatalogDbContext _context;
    public UnitOfWork(CatalogDbContext context) => _context = context;

    public Task<int> SaveChangesAsync() => _context.SaveChangesAsync();
}

This is pure boilerplate. DbContext.SaveChangesAsync() already coordinates all tracked changes as a Unit of Work. Adding a wrapper class that calls it achieves nothing. Catalog.API instead calls _context.SaveChangesAsync() directly inside the repository implementation — the Unit of Work is being used, just without a redundant interface on top of it.

Booking.API's IBookingRepository.SaveAsync() is the same idea expressed slightly differently — it hides the EF Core call behind the interface, but for an architectural reason (Domain cannot reference EF Core), not to introduce a separate Unit of Work pattern.

Interview answer — "What is the Unit of Work pattern and does EF Core implement it?"
"Unit of Work tracks all changes made during a business operation and flushes them to the database in one transaction. EF Core's DbContext implements it natively — the change tracker records every mutation, and SaveChangesAsync commits them all atomically. I don't add a separate IUnitOfWork wrapper on top because DbContext already is one. Adding a wrapper just creates boilerplate that delegates straight through. Where I do use a repository interface — like IBookingRepository in Booking.API — it's to enforce the Clean Architecture boundary: Domain must not reference EF Core, so the interface lives in Domain and the EF Core implementation lives in Infrastructure."
Interview answer — "Do you agree with Nick Chapsas that you should never use a repository with EF Core?"
"Mostly yes, in simple cases. DbContext is already a repository and unit of work — adding IRepository on top is wrapping an abstraction with another abstraction, and you lose EF Core features like projections, AsNoTracking, and IQueryable composability. The 'swap the database' argument rarely holds up in practice. But I'd use the repository interface when there's a genuine architectural reason — like Clean Architecture's dependency rule, where Domain cannot reference EF Core directly, or when the same interface needs multiple implementations across different storage backends. The key is that the interface earns its existence through a real constraint, not just as a habit from pre-EF patterns."
