Section 1

Types — class, interface, record

C# is a nominally typed language. Every value has a declared type. The main type kinds you'll use are class, interface, record, and struct.

class

A reference type. Instances live on the heap. Two variables can point to the same object. Classes support inheritance, encapsulation, and mutable state.

Booking.Domain/Booking.cs
public class Booking
{
    public Guid Id { get; private set; }
    public string Status { get; private set; } = "Pending";

    private Booking() { }     // private constructor — must use Create()

    public static Booking Create(string eventId, string seatId, string userId)
        => new Booking { Id = Guid.NewGuid(), ... };
}

interface

A contract — a list of members a type must implement. Contains no implementation (except default interface methods, rarely used). The cornerstone of dependency inversion: code against the interface, swap the implementation.

Booking.Domain/IBookingRepository.cs
public interface IBookingRepository
{
    Task AddAsync(Booking booking);
    Task SaveAsync();
    Task<Booking?> GetByIdAsync(Guid id);
    Task<List<Booking>> GetPendingBookingsOlderThan(DateTime cutoff);
    Task<List<Booking>> GetConfirmedUnpublishedBookings();
}

BookingRepository in Infrastructure implements this. Program.cs and workers only reference IBookingRepository — they never know it's EF Core underneath.

record

A reference type optimised for immutable data. Equality is value-based (two records with the same properties are equal). Automatically generates ToString(), Equals(), and a copy constructor.

Booking.API/BookingEvent.cs
// Positional record — constructor + properties generated automatically
public record BookingEvent(string EventId, string SeatId, string UserId);

Section 2

Access Modifiers & Properties

Access modifiers control what code can see and change a member. In DDD they enforce the aggregate root: nothing outside the class can assign to Status.

ModifierVisible to
publicEveryone — any assembly, any code
privateOnly the declaring class itself
protectedThe class and any subclass
internalAny code in the same assembly (.dll)
private protectedSubclasses in the same assembly

private set on properties

A property can have different access on its getter and setter. private set means any code can read, but only the class itself can write. This is what makes the aggregate root pattern enforceable at compile time.

Anemic — no protection

public string Status { get; set; }

// Anyone can do this — no rules
booking.Status = "Confirmed";
booking.Status = "Banana"; // compiles fine

Aggregate Root — compiler enforced

public string Status { get; private set; }

// This is a compile error outside the class:
booking.Status = "Confirmed"; // ❌ CS0272
booking.Confirm();             // ✅ only valid path
💡
EF Core bypass EF Core uses reflection to set private set properties when materialising objects from the database. Reflection bypasses all access modifiers — private, protected, all of them. EF Core is the framework, not your application code, so this is intentional. Your code still can't bypass it — only the framework can.

init — write-once properties (C# 9+)

init allows a property to be set during object initialisation but never again. Unlike private set, it can be set from outside the class — but only in an object initialiser or constructor.

concept
public record BookingEvent
{
    public string EventId { get; init; } = string.Empty;
}

var evt = new BookingEvent { EventId = "evt-1" }; // ✅ object initialiser
evt.EventId = "evt-2";                              // ❌ compile error after init
Section 3

Constructors & Object Creation

How an object is created is as important as what it does. The static factory method pattern replaces new with an expressive, rules-enforcing entry point.

Private constructor

A constructor marked private prevents any external code from calling new Booking(). All creation must go through the designated path.

Booking.Domain/Booking.cs
private Booking() { }   // EF Core calls this via reflection
                           // Your code cannot call it at all

Static factory method

A public static method on the class that creates and returns an instance. It has a name (Create), enforces valid initial state, and is the only path into the object.

Booking.Domain/Booking.cs
public static Booking Create(string eventId, string seatId, string userId)
{
    return new Booking          // new is only called from inside the class
    {
        Id        = Guid.NewGuid(),
        EventId   = eventId,
        SeatId    = seatId,
        UserId    = userId,
        Status    = "Pending",    // every booking starts Pending — always
        CreatedAt = DateTime.UtcNow
    };
}
new Booking(...)Booking.Create(...)
Has a nameNo — just newYes — Create expresses intent
Enforces initial stateNo — you set fields yourselfYes — Status is always "Pending"
Can be prevented externallyNo — can't make new privateYes — private constructor blocks new
Can return nullNoYes — useful for conditional creation

Object initialiser

Sets properties by name after construction, in braces. Only works with public set or init properties.

concept
var booking = new Booking
{
    Id        = Guid.NewGuid(),
    EventId   = "evt-coldplay",
    Status    = "Pending"
};
// Equivalent to setting each property on the line after new
Section 4

Methods — expression-bodied, extension, async

Expression-bodied members (C# 6+)

When a method or property is a single expression, replace the body with =>. No { return ...; } needed. Clean for one-liners.

Booking.Infrastructure/BookingRepository.cs
// Method body
public async Task<Booking?> GetByIdAsync(Guid id) =>
    await _context.Bookings.FindAsync(id);

// Property getter
public DbSet<Booking> Bookings => Set<Booking>();

// Constructor shorthand
public BookingRepository(BookingDbContext context) => _context = context;

Extension methods

Add methods to existing types without modifying or subclassing them. Defined as static methods in a static class, with this T as the first parameter. Called as if they're instance methods on T.

Booking.Infrastructure/InfrastructureExtensions.cs
public static class InfrastructureExtensions
{
    // 'this IServiceCollection' makes this an extension on IServiceCollection
    public static IServiceCollection AddBookingInfrastructure(
        this IServiceCollection services, string connectionString)
    {
        services.AddDbContext<BookingDbContext>(opts => opts.UseSqlServer(connectionString));
        services.AddScoped<IBookingRepository, BookingRepository>();
        return services;
    }
}

// Called as if it belongs to IServiceCollection:
builder.Services.AddBookingInfrastructure(connectionString);
💡
Where they're used everywhere in .NET AddDbContext<T>(), AddScoped<T>(), MapPost(), ToListAsync(), UseSqlServer() — these are all extension methods. The pattern of returning this (the same instance) is called a fluent API.
Section 5

Namespaces & Using Directives

Namespaces organise types into named scopes. using imports a namespace so you can refer to its types by short name instead of fully qualifying them.

File-scoped namespace (C# 10+)

One namespace per file — the old curly-brace syntax needed a level of indentation for the entire file. File-scoped applies to everything in the file.

Old syntax

namespace Booking.Domain
{
    public class Booking
    {
        // indented inside namespace
    }
}

File-scoped (C# 10+)

namespace Booking.Domain; // applies to whole file

public class Booking
{
    // no extra indentation
}

using directive vs global namespace

Types in different namespaces within the same assembly still need using to be referenced by short name. Program.cs uses top-level statements — the C# compiler places that code in the global (root) namespace. Workers declared in namespace Booking.API; require using Booking.API; in Program.cs, even though they're in the same project.

Booking.API/Program.cs
using Booking.API;           // OutboxRelayWorker, BookingExpiryWorker
using Booking.Domain;        // IBookingRepository, Booking
using Booking.Infrastructure; // AddBookingInfrastructure()
using StackExchange.Redis;
using Serilog;

// Top-level statements: no namespace declaration
// This code is in the global namespace.
var builder = WebApplication.CreateBuilder(args);

Implicit usings (.NET 6+)

The SDK automatically generates using directives for the most common namespaces (System, System.Collections.Generic, System.Threading.Tasks, etc.) so you don't have to write them in every file. Enabled via <ImplicitUsings>enable</ImplicitUsings> in the csproj.

Fully qualified names

When two namespaces define a type with the same name, you can fully qualify to disambiguate:

Booking.API/Program.cs
// Both Booking.API and Booking.Domain exist — qualify the one we mean
var booking = Booking.Domain.Booking.Create(eventId, seatId, userId);

Generics

Generics allow types and methods to be parameterised over other types. Write once, work safely with any type — without casting.

Generic types and methods

concept
// List<T> — T is the type parameter, filled in at the call site
List<Booking> bookings = new();       // T = Booking
List<string>  names    = new();       // T = string

// Task<T> — T is the result type of the async operation
Task<Booking?> GetByIdAsync(Guid id);   // returns a Booking (or null)
Task          SaveAsync();              // returns nothing

// Generic method
services.AddScoped<IBookingRepository, BookingRepository>();
//          TService = IBookingRepository
//          TImplementation = BookingRepository

Constraints

Type parameters can be constrained to require a type that implements an interface, inherits a class, or has a parameterless constructor.

concept
// T must implement IDisposable
public static void UseAndDispose<T>(T obj) where T : IDisposable
{
    obj.Dispose();
}

// AddDbContext uses: where TContext : DbContext
services.AddDbContext<BookingDbContext>(...);
Section 7

Async / Await

Async/await lets the thread yield while waiting for I/O (database, network, Redis) and resume when the result is ready. No thread is blocked — it can serve other requests instead.

Task, Task<T>, async, await

Booking.Infrastructure/BookingRepository.cs
// async: marks the method as asynchronous
// Task<List<Booking>>: returns a list when done
public async Task<List<Booking>> GetPendingBookingsOlderThan(DateTime cutoff)
{
    // await: pause here, release the thread, resume when ToListAsync is done
    return await _context.Bookings
        .Where(b => b.Status == "Pending" && b.CreatedAt < cutoff)
        .ToListAsync();
}

// Task (no T): no return value, just signals completion
public async Task SaveAsync() =>
    await _context.SaveChangesAsync();
Return typeMeaningExample
voidFire and forget — caller can't await. Avoid in async code.Event handlers only
TaskAsync, no return value. Caller can await completion.SaveAsync()
Task<T>Async, returns T when done.GetByIdAsync() returns Task<Booking?>
ValueTask<T>Like Task<T> but avoids heap allocation when result is already available.High-perf hot paths

CancellationToken

A signal that the caller wants to cancel the operation. Workers receive it from the host — when the app shuts down, the host signals the token and workers stop gracefully.

Booking.API/BookingExpiryWorker.cs
// stoppingToken is cancelled when the host shuts down
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        // Passes token to Delay — throws OperationCanceledException on shutdown
        await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
    }
}
⚠️
Don't call .Result or .Wait() on a Task Blocking on an async method (task.Result, task.Wait()) can deadlock in ASP.NET Core. Always await — propagate async all the way up.
Section 8

Null Safety

C# 8 introduced nullable reference types. The compiler tracks nullability and warns when you might dereference null. Enabled with <Nullable>enable</Nullable> in the csproj.

Nullable reference types

concept
string  name    = null; // ⚠️ warning: non-nullable assigned null
string? nullName = null; // ✅ explicitly nullable

// Repository: returns Booking or null if not found
Task<Booking?> GetByIdAsync(Guid id);

// Caller must check before using
var booking = await repository.GetByIdAsync(bookingId);
if (booking is null)
    return Results.NotFound();

Null-forgiving operator !

Tells the compiler "I know this isn't null, don't warn." Use only when you're certain — it disables the null check, it doesn't add one.

Booking.API/Program.cs
// GetConnectionString returns string? but we know it's in appsettings
builder.Services.AddBookingInfrastructure(
    builder.Configuration.GetConnectionString("SqlServer")!);

Null-conditional operator ?.

Returns null instead of throwing if the left side is null. Short-circuits the entire chain.

concept
string? upper = user?.Name?.ToUpper();
// If user is null → upper = null
// If user.Name is null → upper = null
// Otherwise → upper = "JOHN"

Null-coalescing operator ??

Returns the left side if not null, otherwise the right side.

concept
string name = user?.Name ?? "Anonymous";
// If user or user.Name is null → "Anonymous"

string? seqUrl = builder.Configuration["SeqUrl"];
if (!string.IsNullOrEmpty(seqUrl))
    configuration.WriteTo.Seq(seqUrl);  // compiler knows seqUrl is non-null here
Section 9

LINQ & Collections

LINQ (Language Integrated Query) provides a uniform syntax for querying any data source: in-memory collections, SQL via EF Core, XML. Most methods accept lambda expressions and are extension methods on IEnumerable<T> or IQueryable<T>.

Lambda expressions

An anonymous function written inline. The => separates parameters from the body.

concept
// (parameter) => expression
b => b.Status == "Pending"
// b is the input, b.Status == "Pending" is the boolean result

// Multiple parameters
(a, b) => a + b

// Block body
b => {
    var cutoff = DateTime.UtcNow.AddMinutes(-5);
    return b.CreatedAt < cutoff;
}

Key LINQ methods in the codebase

Booking.Infrastructure/BookingRepository.cs
// Where — filter rows matching the predicate
_context.Bookings
    .Where(b => b.Status == "Pending" && b.CreatedAt < cutoff)

// ToListAsync — materialise to List<T>, hits the DB
.ToListAsync()

// FindAsync — look up by primary key, checks the change tracker first
_context.Bookings.FindAsync(id)

// Count — number of items in a List
if (expired.Count > 0)
    await repository.SaveAsync();
💡
IEnumerable vs IQueryable Where on an IQueryable<T> (EF Core) builds a SQL query — the filter runs in the database. Where on an IEnumerable<T> (in-memory list) runs in C#, loading all rows first. Always use IQueryable with EF Core for filtered queries.

Section 10

Exception Handling

throw in domain methods

Throwing an exception from a domain method signals a genuine problem — a state transition that violates a business rule. The caller catches it and maps it to an HTTP response.

Booking.Domain/Booking.cs
public void Confirm()
{
    if (Status == "Confirmed") return;         // idempotent
    if (Status != "Pending")
        throw new InvalidOperationException(
            $"Cannot confirm a {Status} booking.");  // caught by endpoint
    Status = "Confirmed";
}

try/catch in endpoints and workers

Booking.API/Program.cs
try
{
    booking.Confirm();
    await repository.SaveAsync();
}
catch (InvalidOperationException ex)     // specific — catches domain errors
{
    return Results.Conflict(new { message = ex.Message });
}

// Workers use a broad catch to prevent crash loops
catch (Exception ex)                     // broad — catches anything
{
    _logger.LogError(ex, "Expiry sweep failed, will retry");
}
💡
Exception hierarchy Exception is the base. SystemException covers CLR errors. ApplicationException was intended for user code — avoid it, just throw specific types like InvalidOperationException, ArgumentException, ArgumentNullException, or your own custom exceptions that inherit from Exception.
Section 11

Dependency Injection

Dependency Injection (DI) is a pattern where an object receives its dependencies from the outside rather than creating them itself. ASP.NET Core has a built-in DI container that manages object creation, wiring, and lifetime automatically.

The two interfaces — IServiceCollection vs IServiceProvider

These are two completely separate roles, used at different times in the application lifecycle.

IServiceCollectionIServiceProvider
Analogy Recipe book — defines what can be made and how Chef — actually makes things on demand
Register You register a service by calling AddScoped / AddSingleton / AddTransient. This writes an entry into the collection: "when someone asks for IBookingRepository, create a BookingRepository with Scoped lifetime." Nothing is constructed yet — registering is just writing a recipe. Not applicable — IServiceProvider does not register. Registration is closed once builder.Build() is called.
Resolve Not applicable — IServiceCollection cannot resolve. It only stores the recipes. Resolving is the act of looking up a registered type and constructing the object (and all its dependencies). When a request arrives and the handler declares IBookingRepository repository, the provider resolves it: looks up the recipe, sees it needs a BookingDbContext, resolves that too, then builds the chain and hands the final object to the handler. The word "resolve" is borrowed from compiler terminology. When you write BookingRepository in C# code, that name is just a symbol — a string of characters. The compiler must resolve it: check the using directives at the top of the file and scan the referenced assemblies (the compiled DLLs the project depends on) to find the declaration — the public class BookingRepository { ... } block. That declaration is the type definition: it is where the compiler learns what fields, methods, and constructors BookingRepository actually has. Once found, every place in your source where you wrote BookingRepository gets linked to that declaration. That linking step — name to declaration — is resolution. Four terms in plain English: symbol — a name written in source code (e.g., BookingRepository); type definition — the public class BookingRepository { ... } declaration that tells the compiler what the type contains; resolution — figuring out which declaration a symbol refers to; call site / bind — the exact line of code where the symbol is used, and the act of connecting that usage to the definition. If the name is ambiguous or missing, you get a CS0246 "type not found" error — the compiler couldn't resolve the symbol. The DI container does the same thing at runtime: IBookingRepository is the symbol, BookingRepository is the definition, and the ServiceDescriptor lookup is the resolution step that binds them.
When used Startup only — before the app starts accepting requests Runtime — during every request and every worker cycle
How accessed builder.Services Via constructor injection, or scope.ServiceProvider in workers
What it holds A list of ServiceDescriptor entries (type → implementation → lifetime) Built from IServiceCollection via builder.Build() — you never call this yourself
In this project builder.Services.AddBookingInfrastructure(...) registers DbContext + IBookingRepository Every HTTP request: container resolves IBookingRepository into the handler automatically
Booking.API/Program.cs — the two phases
// ── PHASE 1: REGISTRATION (IServiceCollection) ─────────────────
// builder.Services IS the IServiceCollection.
// Everything here is just adding entries to a list.
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IConnectionMultiplexer>(redis);
builder.Services.AddBookingInfrastructure(connectionString);   // our extension method
builder.Services.AddHostedService<BookingExpiryWorker>();

// ── PHASE 2: BUILD (IServiceCollection → IServiceProvider) ─────
// builder.Build() freezes the registrations and creates the IServiceProvider.
// After this line you can no longer register services.

// ── PHASE 3: RESOLUTION (IServiceProvider at runtime) ──────────
// The container resolves services automatically for every request.
app.MapPost("/book/{eventId}/{seatId}", async (IBookingRepository repository) => { ... });

app.Run();

ServiceDescriptor — what a registration actually stores

Under the hood, every AddScoped<TService, TImplementation>() call creates a ServiceDescriptor and appends it to the list. It records three things:

concept — what AddScoped does internally
// This call:
services.AddScoped<IBookingRepository, BookingRepository>();

// Creates and stores this descriptor:
new ServiceDescriptor(
    serviceType:        typeof(IBookingRepository),   // what callers ask for
    implementationType: typeof(BookingRepository),     // what gets created
    lifetime:           ServiceLifetime.Scoped         // how long it lives
)

When code asks for IBookingRepository, the provider looks up this entry and creates a BookingRepository with the appropriate lifetime.

Constructor Injection — step by step

Constructor injection is the primary way services are delivered. You declare what you need as constructor parameters; the container reads those parameters via reflection and fulfils them automatically. You never call new BookingRepository(...) yourself.

Booking.Infrastructure/BookingRepository.cs
public class BookingRepository : IBookingRepository
{
    private readonly BookingDbContext _context;

    // The container sees this constructor via reflection.
    // It knows it needs a BookingDbContext — so it resolves one first,
    // then calls this constructor with the resolved instance.
    public BookingRepository(BookingDbContext context) => _context = context;
}

The container resolves the entire dependency tree recursively. When a request comes in for IBookingRepository, the chain looks like this:

dependency resolution chain — what happens inside the container
// HTTP request arrives at /payments/{bookingId}/confirm
// Handler declares: (IBookingRepository repository) → container kicks in

Step 1: "I need IBookingRepository"
        → Lookup: IBookingRepository maps to BookingRepository (Scoped)
        → Need to construct BookingRepository

Step 2: "BookingRepository constructor needs BookingDbContext"
        → Lookup: BookingDbContext registered via AddDbContext() (Scoped)
        → Need to construct BookingDbContext

Step 3: "BookingDbContext constructor needs DbContextOptions<BookingDbContext>"
        → Lookup: registered automatically by AddDbContext() (Singleton)
        → Options already exist — reuse

Step 4: DbContextOptions resolved ✓
        → new BookingDbContext(options) created ✓

Step 5: BookingDbContext resolved ✓
        → new BookingRepository(dbContext) created ✓

Step 6: IBookingRepository resolved ✓
        → injected into handler as 'repository' parameter ✓
💡
You declare what you need — the container figures out how to build it Your code never writes new BookingRepository(new BookingDbContext(...)). You declare IBookingRepository repository as a parameter and the container reconstructs the entire tree. If you add a new dependency to BookingRepository's constructor, the container picks it up automatically — nothing else changes.

How the container resolves — the reflection mechanics

The container is not magic — it uses C# reflection to inspect your types at runtime. Reflection is the ability of code to read its own type metadata: what constructors a class has, what parameters those constructors take, what type each parameter is. The container exploits this to wire up the dependency tree without you writing any glue code.

Here is a simplified version of what GetRequiredService<IBookingRepository>() actually does internally:

pseudo-code — what the container does internally to resolve a type
object Resolve(Type serviceType, IServiceScope scope)
{
    // 1. Look up the ServiceDescriptor for the requested type.
    //    This is a dictionary keyed by service type.
    ServiceDescriptor descriptor = _registrations[serviceType];
    //    e.g. { serviceType: IBookingRepository,
    //           implType:    BookingRepository,
    //           lifetime:    Scoped }

    // 2. Check the lifetime cache — no need to construct twice.
    //    Singleton: check _singletonCache (app-wide)
    //    Scoped:    check scope._scopedCache (per-request)
    //    Transient: skip cache — always construct a new instance
    if (descriptor.Lifetime == Scoped && scope._cache.TryGetValue(serviceType, out var cached))
        return cached;  // already built this request — reuse it

    // 3. Find the implementation type's constructor via reflection.
    //    GetConstructors() returns all public constructors.
    //    The container picks the one with the most parameters it can satisfy.
    ConstructorInfo ctor = descriptor.ImplementationType
        .GetConstructors()                // reflection: reads type metadata
        .OrderByDescending(c => c.GetParameters().Length)
        .First();

    // 4. Resolve each constructor parameter recursively.
    //    GetParameters() returns the parameter list — name, type, position.
    //    For BookingRepository: one parameter of type BookingDbContext.
    ParameterInfo[] parameters = ctor.GetParameters();
    object[] args = parameters
        .Select(p => Resolve(p.ParameterType, scope))  // ← recursion
        .ToArray();
    //    For BookingRepository this recurses into Resolve(BookingDbContext),
    //    which recurses into Resolve(DbContextOptions<BookingDbContext>),
    //    which is already cached as a Singleton — base case, recursion unwinds.

    // 5. Construct the object — equivalent to calling new BookingRepository(dbContext).
    //    Activator.CreateInstance invokes the constructor reflectively.
    //    (Real containers compile this to a delegate at startup for performance.)
    object instance = Activator.CreateInstance(descriptor.ImplementationType, args);

    // 6. Store in the appropriate lifetime cache, then return.
    if (descriptor.Lifetime != Transient)
        scope._cache[serviceType] = instance;

    return instance;
}

Three reflection APIs do all the heavy lifting:

Reflection APIWhat it readsUsed for
Type.GetConstructors() All public constructors of a class Finding which constructor to invoke
ConstructorInfo.GetParameters() Each parameter's name and Type Knowing what types to resolve recursively
Activator.CreateInstance(type, args[]) — Calling the constructor to create the instance
💡
Real containers compile away the reflection overhead Reflection is slow when called millions of times. ASP.NET Core's built-in container and third-party containers like Autofac use Expression.Lambda to compile each constructor call into a typed delegate (Func<IServiceProvider, object>) the first time a type is resolved. Subsequent resolutions call the compiled delegate directly — no reflection overhead. The first request pays the compilation cost; every request after that is near-native speed.

public static IServiceCollection AddBookingInfrastructure(
    this IServiceCollection services, string connectionString)
{

    // It also registers DbContextOptions<BookingDbContext> as Singleton.
    services.AddDbContext<BookingDbContext>(opts => opts.UseSqlServer(connectionString));

IServiceProvider — direct resolution (GetRequiredService)

Sometimes you need to resolve a service manually rather than via constructor injection. GetRequiredService<T>() throws if the service is not registered. GetService<T>() returns null instead.

concept
IServiceProvider provider = ...; // obtained from scope or app

// Throws InvalidOperationException if IBookingRepository not registered
var repository = provider.GetRequiredService<IBookingRepository>();

// Returns null if not registered — use when optional
var optional = provider.GetService<ISomeOptionalService>();

// Resolve all registered implementations of an interface
var all = provider.GetServices<IValidator>(); // returns IEnumerable<IValidator>

Captive dependency — Singleton holding a Scoped service

A Singleton lives for the entire application lifetime. If it holds a reference to a Scoped service, that service never gets disposed — it's "captured" in the Singleton forever. ASP.NET Core detects this at startup and throws.

Broken — captive dependency

// Workers are Singleton.
// IBookingRepository is Scoped.
// This throws at startup:
public BookingExpiryWorker(
    IBookingRepository repository) // ❌
{
    _repository = repository;
    // Scoped service captured in Singleton
    // Same DbContext reused forever
    // Memory leak + stale data
}

Correct — scope per cycle

// Inject IServiceScopeFactory (Singleton-safe).
// Create a fresh scope per worker cycle.
public BookingExpiryWorker(
    IServiceScopeFactory scopeFactory) // ✅
{
    _scopeFactory = scopeFactory;
}

// In ExecuteAsync:
using var scope = _scopeFactory.CreateScope();
var repo = scope.ServiceProvider
    .GetRequiredService<IBookingRepository>();
// New DbContext per cycle. Disposed with the scope.
⚠️
Rule: never inject a shorter-lived service into a longer-lived one Singleton can only hold Singletons. Scoped can hold Scoped or Singleton. Transient can hold anything. Violating this is a captive dependency — the shorter-lived service gets pinned to the longer-lived container and is never properly disposed.
Section 11b

Top-Level Statements & Program.cs

C# 9 introduced top-level statements — a way to write the program entry point without explicitly declaring a Main method or wrapping everything in a class. ASP.NET Core minimal APIs are built on this.

What top-level statements actually are

The compiler generates the boilerplate for you. Behind the scenes, everything in Program.cs becomes the body of a Main method inside a compiler-generated class called <Program>.

Old — explicit Main

public class Program
{
    public static void Main(string[] args)
    {
        var builder =
            WebApplication.CreateBuilder(args);

        builder.Services.AddBookingInfrastructure(...);

        var app = builder.Build();
        app.MapPost("/book", ...);
        app.Run();
    }
}

New — top-level statements (C# 9+)

// No class, no Main, no namespace.
// The compiler generates all of that.
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddBookingInfrastructure(...);

var app = builder.Build();
app.MapPost("/book", ...);
app.Run();

The global namespace

Top-level statement code lives in the global (root) namespace — there is no enclosing namespace declaration. This has one important consequence: types defined in named namespaces within the same project are not automatically in scope. You still need using directives to reference them by short name.

Booking.API/Program.cs
// Program.cs is in the global namespace (no namespace declaration).
// Workers declare: namespace Booking.API;
// Even though they're in the same project, they're in a different namespace.
// Without this using, OutboxRelayWorker and BookingExpiryWorker cannot be found.
using Booking.API;           // OutboxRelayWorker, BookingExpiryWorker
using Booking.Domain;        // IBookingRepository
using Booking.Infrastructure; // AddBookingInfrastructure()

// Top-level code starts here — no wrapping needed
var builder = WebApplication.CreateBuilder(args);

Constraints of top-level statements

  • Only one file per project may contain top-level statements. The compiler generates one Main entry point — two files with top-level statements would create two Main methods.
  • Cannot be inside a namespace. File-scoped namespaces (namespace Foo;) and top-level statements cannot coexist in the same file.
  • Local types are allowed. You can declare classes, records, and interfaces at the bottom of Program.cs — they'll be part of the global namespace.
Booking.API/Program.cs — local type declared at the bottom
// ... all the app.MapPost / app.Run code above ...

app.Run();

// Types declared after app.Run() are still valid — they're in the global namespace.
// BookingEvent record is declared here because it's small and only used in this project.
record BookingEvent(string EventId, string SeatId, string UserId);

WebApplication builder — the startup pipeline

Program.cs follows a strict three-phase structure: configure services → build → configure middleware/endpoints → run.

Booking.API/Program.cs — annotated structure
// ── 1. CREATE BUILDER ─────────────────────────────────────────────
// Sets up configuration (appsettings.json, env vars), logging, hosting.
var builder = WebApplication.CreateBuilder(args);

// ── 2. REGISTER SERVICES (IServiceCollection) ─────────────────────
// builder.Services is the IServiceCollection.
// Anything registered here is available for injection at runtime.
builder.Services.AddSingleton<IConnectionMultiplexer>(redis);
builder.Services.AddBookingInfrastructure(connectionString);
builder.Services.AddHostedService<BookingExpiryWorker>();

// ── 3. BUILD (IServiceCollection → IServiceProvider) ──────────────
// Freezes registrations. Creates the IServiceProvider.
// No new services can be registered after this line.
var app = builder.Build();

// ── 4. REGISTER MIDDLEWARE & ENDPOINTS ────────────────────────────
// app.Use... adds middleware to the request pipeline.
// app.MapPost... registers route handlers.
app.MapPost("/book/{eventId}/{seatId}", async (IBookingRepository repository, ...) => { ... });
app.MapPost("/payments/{bookingId}/confirm", async (IBookingRepository repository, ...) => { ... });

// ── 5. RUN ────────────────────────────────────────────────────────
// Starts the HTTP server. Blocks until the app shuts down.
app.Run();
💡
args is the command-line arguments array WebApplication.CreateBuilder(args) receives the same string[] args that would have been the Main parameter. The compiler passes the generated Main's args through automatically. This is how you can override configuration from the command line (dotnet run --urls http://localhost:5001).
Section 12

Pattern Matching

Pattern matching lets you inspect the shape or value of an expression and branch accordingly. Cleaner than a chain of if (x is SomeType) checks.

is null / is not null

Booking.API/Program.cs
var booking = await repository.GetByIdAsync(bookingId);
if (booking is null)
    return Results.NotFound(new { message = "Booking not found." });

Switch expression (C# 8+)

Returns a value based on matching. Exhaustiveness is checked — the compiler warns if a case is unhandled.

concept
string label = booking.Status switch
{
    "Pending"   => "Awaiting payment",
    "Confirmed" => "Payment received",
    "Expired"   => "Booking lapsed",
    _           => "Unknown"           // discard pattern — catch-all
};

Type patterns

concept
if (ex is InvalidOperationException ioe)
{
    // ioe is now typed as InvalidOperationException
    logger.LogWarning(ioe.Message);
}
Section 13

BackgroundService & Hosted Services

A BackgroundService runs alongside the web server — it starts when the app starts and is cancelled when it shuts down. Used for polling, cleanup sweeps, and outbox relays.

Structure

Booking.API/BookingExpiryWorker.cs
// Inherit BackgroundService — implement one method
public class BookingExpiryWorker : BackgroundService
{
    // Called once by the host. Loop yourself — host doesn't restart it.
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                // ... work ...
            }
            catch (Exception ex)
            {
                // catch here — unhandled exception kills the worker permanently
                _logger.LogError(ex, "Sweep failed, will retry");
            }
            await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
        }
    }
}

Registration

Booking.API/Program.cs
// Registered as a hosted service — host starts it automatically
builder.Services.AddHostedService<OutboxRelayWorker>();
builder.Services.AddHostedService<BookingExpiryWorker>();
⚠️
Workers are registered as Singleton AddHostedService<T> registers T as a Singleton. You cannot inject Scoped services (DbContext, repositories) directly into the constructor — use IServiceScopeFactory and create a scope per cycle instead.
Section 14

Miscellaneous — var, string interpolation, using declarations

var — type inference

The compiler infers the type from the right side. The type is still static — it's just written once.

concept
var booking = await repository.GetByIdAsync(id);
// compiler infers: Booking? booking

var expired = await repository.GetPendingBookingsOlderThan(cutoff);
// compiler infers: List<Booking> expired

String interpolation ($"...")

Embed expressions directly in a string literal. Evaluated at runtime.

Booking.API/Program.cs
var lockKey  = $"seat:lock:{eventId}:{seatId}";
var seatsKey = $"event:{eventId}:seats";

// With formatting
$"Price: {price:C2}"   // currency, 2 decimal places
$"Date: {date:yyyy-MM-dd}"

using declarations (C# 8+)

using var x = ... disposes x at the end of the enclosing scope (closing brace). Cleaner than the old using (var x = ...) { } block.

Booking.API/OutboxRelayWorker.cs
// Old style — requires extra braces and indentation
using (var scope = _scopeFactory.CreateScope()) { ... }

// New style — disposed when the enclosing block ends
using var scope      = _scopeFactory.CreateScope();
using var connection = await factory.CreateConnectionAsync(stoppingToken);
using var channel    = await connection.CreateChannelAsync();
// all three disposed in reverse order at end of try block

Guid and DateTime

Booking.Domain/Booking.cs
Guid Id = Guid.NewGuid();   // generates a new random UUID: "3f2504e0-4f89-..."

DateTime CreatedAt = DateTime.UtcNow;
// Always use UtcNow in server code — no timezone ambiguity

TimeSpan HoldWindow = TimeSpan.FromMinutes(5);
DateTime cutoff     = DateTime.UtcNow - HoldWindow;
// DateTime arithmetic: subtract a TimeSpan to get an earlier DateTime

readonly fields

readonly on a field means it can only be assigned in the constructor or at declaration. Combined with private, it prevents both external and internal reassignment after construction.

Booking.API/BookingExpiryWorker.cs
private readonly IServiceScopeFactory _scopeFactory;
private readonly IConnectionMultiplexer _redis;
private readonly ILogger<BookingExpiryWorker> _logger;

// Set once in constructor — cannot be reassigned anywhere in the class
public BookingExpiryWorker(IServiceScopeFactory scopeFactory, ...)
{
    _scopeFactory = scopeFactory;  // only valid assignment
}

static readonly — class-level constants

Booking.API/BookingExpiryWorker.cs
// One instance shared by all instances of the class
private static readonly TimeSpan HoldWindow = TimeSpan.FromMinutes(5);
// static: belongs to the class, not an instance
// readonly: set once at declaration, never changed
