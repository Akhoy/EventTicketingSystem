1. System Overview

This is a cloud-native event ticketing system — users browse events, reserve seats, and confirm bookings through a payment step. The system is intentionally complex enough to demonstrate real production concerns: concurrency, distributed consistency, reliable messaging, and compensating transactions.

It is built as a set of microservices — independently deployable processes that each own their own data store and communicate through well-defined contracts. No service reaches into another service's database.

3. Services Deep Dive

API Gateway (YARP)

YARP (Yet Another Reverse Proxy) is Microsoft's reverse proxy library. It is the single entry point for all clients. No client ever calls Booking.API or Catalog.API directly — they call the gateway.

Routing rules

  • /catalog/** → strips the prefix and forwards to http://catalog-api:8080
  • /book/** → forwards as-is to http://booking-api:8080
Why a gateway? Without it, clients must know the address of every downstream service. Adding a new service means updating every client. The gateway absorbs that complexity — clients have one address forever. It also becomes the single place to add authentication enforcement, rate limiting, SSL termination, and request logging in the future.
Booking.API

Owns the entire booking lifecycle: reservation → payment confirmation → event publication.

Endpoints

EndpointWhat it does
POST /book/{eventId}/{seatId} Acquires seat lock, decrements capacity, creates Pending booking in SQL
POST /payments/{bookingId}/confirm Flips booking to Confirmed inside a DB transaction

Background Workers

WorkerPoll intervalWhat it does
OutboxRelayWorker5 secondsFinds Confirmed + unpublished bookings, pushes to RabbitMQ, marks PublishedAt
BookingExpiryWorker30 secondsFinds Pending bookings older than 5 minutes, restores Redis counter, releases seat lock, marks Expired

Data stores

  • SQL Server — authoritative source of truth for all bookings. ACID transactions. EF Core manages schema via migrations.
  • Redis — two separate concerns:
    • event:{eventId}:seats — atomic integer counter, decremented on reserve, incremented on release
    • seat:lock:{eventId}:{seatId} — SETNX flag with 5-min TTL, prevents two users booking the exact same seat
Catalog.API

Owns the event catalogue and provides the public-facing seat availability for browsing.

Endpoints

EndpointWhat it does
GET /itemsReturns all events with their current AvailableSeats from MongoDB

WorkerWhat it does

  • MongoDB — document store for the event catalogue. AvailableSeats here is the display count — eventually consistent, not the booking gate.
  • Redis — Catalog.API seeds the initial seat counters on startup. It owns the initial values; Booking.API owns the decrements thereafter.

4. The Booking Flow — Step by Step

Step 1: User books a seat

POST /book/evt-coldplay/A1

Seat lock (SETNX) — Redis SET seat:lock:evt-coldplay:A1 {userId} EX 300 NX. The NX flag means "only set if not exists". If someone else holds this exact seat, the command returns false and the request is rejected immediately with 409 Conflict.
Why? Two users could both pass the capacity check simultaneously. The seat lock prevents two different people from booking the same physical seat A1 for the same event.
Capacity guard (DECR) — Redis DECR event:evt-coldplay:seats. This is atomic — Redis processes it as a single operation. The returned value is the new count. If it's below 0, the event is sold out; we immediately INCR to restore the counter and release the seat lock.
Why Redis, not SQL? SQL would require a SELECT + UPDATE under a lock — two round trips, heavy under load. A Redis DECR is a single atomic instruction, typically sub-millisecond. Under 10,000 concurrent booking attempts, this does not race.
Persist Pending booking — SQL Server INSERT. Wrapped in try/catch: if SQL fails after the Redis DECR, we roll back both the counter (INCR) and the seat lock (DEL) before returning an error.
Why? If SQL throws and we don't roll back Redis, the seat counter is permanently one lower than reality. That seat is silently lost — neither bookable nor shown as available. The rollback closes that gap.
Return bookingId to the client. The booking is Pending. Seat is held for 5 minutes.

Step 2: User pays

POST /payments/{bookingId}/confirm

Load the booking from SQL. Validate it exists and is still Pending. If already Confirmed or Expired, reject with 409.
Open a database transaction, flip Status to Confirmed, commit.
Why a transaction? In future when publishing to a payment provider runs in the same unit of work, the transaction ensures both succeed or both roll back.

Step 3: OutboxRelayWorker publishes the event

Every 5 seconds, the worker queries SQL: WHERE Status = 'Confirmed' AND PublishedAt IS NULL.
For each unpublished booking, it serialises a BookingEvent DTO and publishes to the ticket_events fanout exchange in RabbitMQ with Persistent = true.
Why not publish directly from the /confirm endpoint? If RabbitMQ is temporarily down, a direct publish would fail — and the booking is already confirmed. Do you return an error? Roll back? The outbox sidesteps this entirely: confirm succeeds, the event sits in SQL, and the relay keeps retrying until RabbitMQ recovers. Nothing is lost.
Mark PublishedAt = UtcNow and save. The next poll cycle skips this booking.

Step 4: CatalogSyncWorker updates the read model

CatalogSyncWorker receives the message from its catalog_sync_queue (bound to the fanout). It deserialises the BookingEvent, extracts the EventId.
MongoDB: db.Tickets.UpdateOne({ EventId: "evt-coldplay" }, { $inc: { AvailableSeats: -1 } })
Why decrement MongoDB at all if Redis already guards capacity? Redis is the authoritative gate for booking decisions. MongoDB is the display layer — what users see on the catalogue page. Without syncing MongoDB, the "seats available" count shown to browsers would never decrease no matter how many bookings occurred.
ACK the message. RabbitMQ removes it from the queue. If the handler throws before ACK, RabbitMQ redelivers.

Step 5: Abandoned checkout — BookingExpiryWorker compensates

Every 30 seconds, the worker queries SQL: WHERE Status = 'Pending' AND CreatedAt < (Now - 5 min).
For each expired booking: INCR the Redis seat counter, DEL the seat lock, set Status = Expired, save to SQL.
Why? When a user abandons checkout, the Redis counter has already been decremented and the seat lock set. Without this worker, those resources are permanently consumed — the event appears sold out even though no one bought the seat. This is the compensating transaction: it undoes the side effects of an operation that was never completed.

5. Design Patterns Employed

Outbox Pattern

Pattern

The Bookings SQL table doubles as a message outbox. The PublishedAt column is the outbox marker — null means the message hasn't been sent yet. The OutboxRelayWorker is the relay that polls and publishes.

This implements at-least-once delivery: if the relay crashes mid-publish, on restart it finds the same rows still with PublishedAt = null and retries. Consumers must be idempotent.

The pain it solves: If you publish directly from the HTTP handler and the broker is down, you must choose: fail the user's request (bad UX), or silently lose the event (data loss). The outbox makes durability the database's problem — SQL is already the source of truth.

Read Model (CQRS) — not yet a true Materialized View

Pattern Enhancement Pending

MongoDB's AvailableSeats is a denormalized derived field — a read model kept eventually consistent by CatalogSyncWorker. The authoritative state for seat availability lives in Redis (booking gate) and SQL (booking records). MongoDB holds a copy shaped for catalogue browsing.

This is not a Materialized View in the strict database sense. A true Materialized View is a separate schema object — a pre-computed query result across multiple sources, rebuilt on change. For example, a proper EventSummaries collection would aggregate confirmed bookings, pending holds, and revenue across Redis + SQL + MongoDB into one document optimised for display: 

// What a proper Materialized View looks like (not yet built)
{
  "EventId": "evt-coldplay",
  "EventName": "Coldplay - Spheres",
  "AvailableSeats": 1985,
  "ConfirmedBookings": 15,
  "PendingHolds": 3,
  "RevenueToDate": 1800.00
}
The pain it solves (intent is right, implementation is partial): Without any read model, every GET /items would need to query Redis for every event's counter — many round trips per page load. The read model trades write complexity (keeping it in sync) for read simplicity (one MongoDB query). The gap: AvailableSeats is just one field on an existing document, not a purpose-built projection across multiple sources. The Materialized View pattern is on the enhancement backlog.

Compensating Transaction

Pattern

The BookingExpiryWorker is a compensating transaction — it undoes the side effects of an incomplete operation. When a user starts a booking (decrement counter, lock seat) but never pays, the worker reverses those effects (increment counter, delete lock).

This is the microservices alternative to a database ROLLBACK. Because Redis and SQL are two separate data stores, there is no single transaction spanning both. The compensation is the manual equivalent.

Relation to Saga pattern: A formal Saga is a sequence of named compensating steps triggered by failure at any point in a distributed workflow. This worker is saga-adjacent — it compensates the booking reservation step specifically. A full Saga would handle payment failures, notification failures, etc.

Repository Pattern

Pattern SOLID — D

Catalog.API uses ICatalogRepository (interface) implemented by CatalogRepository (MongoDB). The application layer (Program.cs, CatalogSyncWorker) depends on the interface, not the concrete MongoDB class. 

// Catalog.API depends on this — knows nothing about MongoDB
public interface ICatalogRepository {
    Task DecrementAvailableSeatsAsync(string eventId);
    Task<List<CatalogItem>> GetAllAsync();
    Task SeedDataAsync(IEnumerable<CatalogItem> items);
}
The pain it solves (Dependency Inversion): If you want to swap MongoDB for Postgres, you write a new class and swap one registration in DI. No other code changes. Without this abstraction, MongoDB driver calls are scattered everywhere and a data store change becomes a full rewrite.

Fanout Exchange (Publish/Subscribe)

Pattern

RabbitMQ's ticket_events exchange is a fanout: it copies every message to every queue bound to it. Today, only catalog_sync_queue is bound. Tomorrow, you bind notification_queue and analytics_queue — the publisher never changes.

The pain it solves: With a direct exchange (point-to-point), adding a new consumer means changing the publisher to know about the new destination. Fanout inverts this — consumers self-register. The publisher is completely decoupled from who's listening.

CQRS — Read/Write Store Separation (partial)

Pattern Enhancement Pending

CQRS (Command Query Responsibility Segregation) means the write path and read path use separate models. This system implements the data side of CQRS:

  • Write path — POST /book, POST /payments/confirm write to Redis + SQL Server (authoritative stores)
  • Read path — GET /items reads from MongoDB (a separate store shaped for display)
  • The two stores are kept in sync asynchronously via CatalogSyncWorker

What is not yet implemented is the code side of CQRS — explicit Command and Query objects with separate handlers (typically via MediatR): 

// Not yet implemented — what formal CQRS looks like
public record CreateBookingCommand(string EventId, string SeatId, string UserId);
public class CreateBookingCommandHandler : IRequestHandler<CreateBookingCommand, Guid> { ... }

public record GetCatalogItemsQuery();
public class GetCatalogItemsQueryHandler : IRequestHandler<GetCatalogItemsQuery, List<CatalogItem>> { ... }

Currently, all logic lives inline in Program.cs endpoints. The formal command/query separation comes with the DDD + Clean Architecture restructure (on the backlog).

The pain it solves: Without CQRS, adding a read replica or pointing reads at a cache requires rewriting the same endpoint that also handles writes. With explicit command/query handlers, you swap the query handler's data source (MongoDB → Redis → read replica) without touching the write handler at all. Read and write scaling become independent decisions.

Atomic Counter + SETNX Lock

Pattern Concurrency

Two independent Redis guards protect the booking path:

// Guard 1 — specific seat lock (SETNX = SET if Not eXists)
bool locked = await cache.StringSetAsync(lockKey, userId,
    TimeSpan.FromMinutes(5), When.NotExists);

// Guard 2 — event capacity (atomic decrement)
long remaining = await cache.StringDecrementAsync(seatsKey);
if (remaining < 0) { /* sold out — roll back */ }

These address two distinct failure modes: two users booking the same seat (seat lock), and total overselling across all seats (capacity counter).

Why two guards? The capacity counter alone prevents overselling but allows two people to "book" seat A1 simultaneously (one would get a meaningless booking). The seat lock alone prevents duplicate A1 bookings but doesn't cap total bookings. You need both.

6. CAP Theorem in Practice

CAP Theorem states a distributed system can only guarantee two of three properties: Consistency (all reads see the latest write), Availability (every request gets a response), and Partition Tolerance (the system works despite network failures). Since partitions are inevitable, the real choice is CP vs AP per component.

This system makes different CAP trade-offs for different data stores — intentionally, based on the business consequence of getting it wrong.

StoreCAP ChoiceBusiness Reason
Redis (seat counter) CP If Redis is unavailable, /book returns an error. We choose to be unavailable rather than risk selling the same seat twice. Overselling a concert ticket is unrecoverable — you cannot un-sell a seat.
MongoDB (catalogue) AP If CatalogSyncWorker is behind, MongoDB shows "120 seats" when Redis says "115". The catalogue stays available with stale data. A user seeing a slightly wrong count is acceptable; a user unable to browse the catalogue is not.
SQL Server (bookings) CP ACID guarantees. If SQL is unavailable, booking fails. The Booking record is the financial source of truth — consistency is non-negotiable.
RabbitMQ (event bus) AP (via outbox) If RabbitMQ is down, the outbox absorbs the messages. Delivery is delayed (temporarily unavailable from the consumer's perspective) but never lost. Eventual consistency.
Interview talking point: "I made different CAP trade-offs per data store based on the cost of inconsistency. Redis and SQL are CP because the booking domain cannot tolerate inconsistency — money is involved. MongoDB and RabbitMQ are AP because slightly stale catalogue data or delayed event delivery is a business-acceptable trade-off."

7. Key Technical Decisions & Why

Database-per-Microservice

Booking.API owns SQL Server. Catalog.API owns MongoDB. Neither service queries the other's database. Cross-service data flows only through RabbitMQ messages.

Why? A shared database is the tightest coupling possible — schema changes in one service break another. Each service must be deployable and scalable independently. That requires full data ownership. The cost: you cannot write a JOIN across services. You design around that with events and materialized views.

Polyglot Persistence

SQL Server for transactional bookings (ACID, strong consistency, relational queries). MongoDB for event catalogue (flexible document model, easy to add fields, scales reads horizontally). Redis for real-time counters and locks (sub-millisecond atomic operations, no SQL locking overhead).

Why not just use one database for everything? Each problem has a different shape. A seat counter under concurrent load is not a SQL problem — it's an atomic increment problem, and Redis solves it natively. A catalogue with varying event metadata is not a great relational fit — MongoDB's document model is more natural. Using the right tool per problem is what polyglot persistence means.

YARP as API Gateway

Microsoft's YARP library configured entirely through appsettings.json — no code required for basic routing.

Why YARP over NGINX or a custom proxy? YARP is a .NET library, not a separate process. It runs in the same ASP.NET Core pipeline, benefits from the same DI container, and is configurable at runtime without a sidecar container. For a .NET shop it is the lowest-friction choice.

BackgroundService for Workers

All workers (OutboxRelayWorker, BookingExpiryWorker, CatalogSyncWorker) implement ASP.NET Core's BackgroundService base class and run as hosted services within the same process as the HTTP server.

Why co-locate workers with the API? At this scale, a separate process for each worker adds operational overhead (more containers, more networking, more monitoring) for no benefit. The BackgroundService lifecycle is managed by the ASP.NET host — it starts on startup and gets a CancellationToken on shutdown, so workers stop gracefully.

IServiceScopeFactory in Background Workers

Workers use IServiceScopeFactory.CreateScope() to resolve BookingDbContext rather than injecting it directly. 

using var scope = _scopeFactory.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<BookingDbContext>();
Why? DbContext is a scoped service (one per HTTP request). BackgroundService is a singleton — it lives for the application's entire lifetime. You cannot inject a scoped service into a singleton (the scoped instance would be captured forever, causing stale state and thread-safety issues). Creating a scope per loop iteration gives the worker a fresh DbContext on each cycle.

BookingEvent as a Duplicated DTO

Both Booking.API and Catalog.API define their own BookingEvent record with identical fields. They are not in a shared library.

Why not share the DTO in a common project? A shared library couples two independently deployable services — a field rename in the shared project forces both services to be updated and redeployed simultaneously. With duplicated DTOs, each service owns its deserialization. The contract is enforced by the wire format (JSON field names), not by a shared type. This is the microservices "share schema, not types" principle.

Serilog → Seq for Structured Logging

Serilog writes structured JSON log entries (not plain text strings). Seq is a local log aggregation server that indexes structured logs and allows querying by property.

Why structured logging? A plain text log entry like "Booking created for seat A1" requires regex to extract the seat. A structured entry { "SeatId": "A1", "EventId": "evt-coldplay" } is queryable by field. When you have three services each writing logs, searching for all log lines related to one booking across all services by BookingId becomes a Seq query, not a grep through files.
