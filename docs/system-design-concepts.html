1. CAP Theorem

Any distributed system can only guarantee two of three properties simultaneously:

LetterPropertyPlain English
CConsistencyEvery read returns the most recent write. No stale data.
AAvailabilityEvery request gets a response. System never refuses.
PPartition ToleranceSystem keeps working even when nodes can't talk to each other.
P is not optional. Networks fail in every real distributed system. So the real choice is always CP or AP — never CA.

CP — Consistency + Partition Tolerance

When a partition occurs, the system rejects requests rather than risk returning stale or inconsistent data. Availability is sacrificed.

Your example: Redis seat counter. If Redis is unreachable, /book returns an error. Better to be down than to oversell a seat.

AP — Availability + Partition Tolerance

When a partition occurs, the system keeps responding — possibly with stale data. Consistency is sacrificed.

Your example: MongoDB AvailableSeats. CatalogSyncWorker may be behind. GET /items still responds with last known values. A slightly wrong seat count on the browse page is acceptable.

Your system's CAP trade-offs

ComponentChoiceWhy
Redis (seat counter)CPOverselling is unrecoverable. Downtime is recoverable.
MongoDB (catalogue)APStale seat count on browse page is acceptable. Unavailability is not.
SQL Server (bookings)CPACID. Source of financial truth.
RabbitMQ + OutboxAPMessages delayed if broker down, never lost. Eventual consistency.
Interview answer: "I made different CAP trade-offs per data store based on the cost of being wrong. Redis and SQL are CP — overselling a seat is unrecoverable so consistency wins. MongoDB is AP — a slightly stale seat count on a browse page is a business-acceptable trade-off. Availability wins there."

2. ACID

ACID is a set of four guarantees a database makes about transactions. A transaction is a group of operations that must all succeed or all fail together.

A — Atomicity

All or nothing. Every operation in a transaction either fully commits or fully rolls back. No half-written state ever exists.

Your example:

using var transaction = await db.Database.BeginTransactionAsync();
booking.Confirm();
await db.SaveChangesAsync();
await transaction.CommitAsync(); // if this fails, nothing was written

If SaveChangesAsync throws, the booking stays Pending. You never get a half-confirmed booking.

Real world: Bank transfer — debit Account A, credit Account B. If the server crashes between them, atomicity guarantees the debit is rolled back. Nobody loses money.

C — Consistency

The database moves from one valid state to another. Rules and constraints always hold.

This C is different from CAP's C. ACID consistency = data follows defined rules (constraints, foreign keys). CAP consistency = every read sees the latest write across distributed nodes. Same letter, completely different meaning.

I — Isolation

Concurrent transactions don't see each other's intermediate state. Controlled by isolation levels (see section 4).

Your example: Two users confirming bookings simultaneously. Transaction 1's half-written state is invisible to Transaction 2.

D — Durability

Once committed, data survives a crash. Written to disk, not just memory.

SQL Server vs Redis: SQL Server writes to a transaction log on disk before confirming a commit. Redis keeps data in memory by default — a restart wipes it. That's why your seat counters need to be repopulated from MongoDB on Redis restart, but your booking records survive SQL Server restarts unchanged.

3. BASE

BASE is the opposite philosophy to ACID. NoSQL databases like MongoDB adopt it.

LetterPropertyPlain English
BBasically AvailableSystem responds most of the time, even if data is stale or partial
SSoft StateState can change over time without a direct write — as sync catches up
EEventually ConsistentGiven enough time, all nodes converge to the same value

Your example: MongoDB AvailableSeats. After a booking is confirmed, CatalogSyncWorker asynchronously decrements the MongoDB count. For 10-30 seconds, MongoDB might show "2000 seats" when Redis says "1995". That's soft state — it's changing without a direct write, as the worker catches up. Eventually both agree. That's BASE.

Interview answer: "My MongoDB catalogue is BASE. AvailableSeats is eventually consistent — it lags behind Redis by however long CatalogSyncWorker takes to process the message. That's acceptable because it's a display value. The booking gate is Redis, which is strongly consistent. I use ACID where money is involved, BASE where approximate data is acceptable."

4. Isolation Levels

Isolation levels control how much concurrent transactions can "see" each other. Higher isolation = more protection, less concurrency, lower throughput.

The Three Problems

Dirty Read

You read data another transaction has written but not yet committed. That transaction might roll back — meaning you read data that never officially existed.

Transaction 1: UPDATE booking SET status = 'Confirmed'  -- not committed yet
Transaction 2: SELECT status FROM booking → 'Confirmed'  -- dirty read
Transaction 1: ROLLBACK                                   -- never happened
-- Transaction 2 acted on data that never existed

Your system impact: OutboxRelayWorker could publish a booking to RabbitMQ, then the HTTP handler rolls back. Catalog.API decrements seats for a booking that never existed.

Non-Repeatable Read

You read the same row twice in one transaction and get different values because another transaction updated it in between.

Transaction 1: SELECT status → 'Pending'
Transaction 2: UPDATE status = 'Confirmed' → COMMIT
Transaction 1: SELECT status → 'Confirmed'  -- different value, same transaction

Same transaction. Same row. Different value. Your logic is now inconsistent with itself.

Phantom Read

You query a set of rows twice in one transaction and get different rows because another transaction inserted or deleted rows in between.

Transaction 1: SELECT COUNT(*) FROM Bookings WHERE EventId = 'evt-coldplay' → 150
Transaction 2: INSERT INTO Bookings (EventId...) VALUES ('evt-coldplay') → COMMIT
Transaction 1: SELECT COUNT(*) FROM Bookings WHERE EventId = 'evt-coldplay' → 151

A new row "appeared" — a phantom. Same query, different result set.

Difference from non-repeatable read: Non-repeatable = same row, value changed. Phantom = same query, different rows returned.

Isolation Levels — What They Prevent

LevelDirty ReadNon-Repeatable ReadPhantom ReadCost
Read Uncommitted❌ Not prevented❌❌Maximum concurrency
Read Committed✅ Prevented❌❌Low — SQL Server default
Repeatable Read✅✅ Prevented❌Medium — locks rows you read
Serializable✅✅✅ PreventedMaximum — kills throughput
Why SQL Server defaults to Read Committed: "It prevents dirty reads — the most dangerous problem — without the throughput cost of Serializable. Serializable locks every transaction so only one runs at a time. Under 50,000 concurrent users that means everything queues behind locks. Read Committed is the pragmatic middle ground."

5. Consistency Models

Between "always right" and "eventually right" there is a spectrum. Different parts of your system sit at different points on it.

ModelMeaningYour example
Strong Consistency Every read returns the latest write. Always. No exceptions. Redis DECR — after it executes, every read of that key sees the decremented value immediately
Read-Your-Writes You always see your own writes immediately. Others might not yet. After confirming a booking, you see "Confirmed" immediately. Another user querying might briefly see "Pending"
Causal Consistency Causally related operations are seen in the correct order by everyone. RabbitMQ — messages within a queue are consumed in order. The cause always appears before the effect.
Eventual Consistency Will converge to the correct value. Timing not guaranteed. MongoDB AvailableSeats — lags behind Redis by seconds. DNS propagation — lags by up to 48 hours.

6. Redis Atomicity & Distributed Counter

What is a Distributed Counter?

A counter that must be accurate across multiple processes running simultaneously. If Booking.API runs as 3 pods, all three pods need to decrement the same count. The counter must live somewhere shared — not in any pod's memory.

Pod 1 → Redis DECR event:evt-coldplay:seats
Pod 2 → Redis DECR event:evt-coldplay:seats  // same key, different pod
Pod 3 → Redis DECR event:evt-coldplay:seats  // all three hit the same Redis key

What is Interleaving?

In a multi-threaded system, the CPU can switch between threads mid-operation. Each "decrement" is actually three steps: read → subtract → write. Interleaving means those steps from different threads mix together:

Thread 1: reads counter = 100
Thread 2: reads counter = 100   // switched before Thread 1 wrote
Thread 1: writes 99
Thread 2: writes 99             // both subtracted 1, counter only moved by 1
// Two bookings happened. One seat consumed. Oversold.

Locks prevent interleaving by forcing threads to take turns. But locks under high concurrency cause contention — threads queue, throughput drops.

Why Redis Doesn't Need Locks

Redis is single-threaded for command execution. Every command runs to completion before the next one starts. No interleaving is possible. No locks needed.

// All three pods fire DECR simultaneously
// Redis processes them one at a time:
Pod 1 DECR → 1999  (complete)
Pod 2 DECR → 1998  (complete)
Pod 3 DECR → 1997  (complete)
// No race condition. No interleaving. No locks.
Interview phrasing: "Redis achieves atomicity through single-threaded command execution, not locking. Every command runs to completion before the next starts. No interleaving is possible so no locks are needed."

Why Single-Threaded Isn't Slow

Redis commands execute in microseconds (millionths of a second). At 50,000 requests per second, that's 50ms of actual work per second. Redis handles over 1 million operations per second on a single thread.

The actual bottleneck is always the network round trip — typically 0.5–2ms. The command itself takes 0.001ms. Single-threaded execution is not the constraint.

Compare to SQL Server: acquire row lock → read → write → release lock → write to transaction log → flush to disk = 5–10ms minimum. Redis: 0.001ms.

The Two Guards in Your System

GuardRedis CommandWhat it prevents
Capacity gate DECR event:{eventId}:seats Total overselling — more bookings than seats exist
Seat lock SET seat:lock:{eventId}:{seatId} NX EX 300 Two users booking the exact same physical seat simultaneously

Why both? The capacity gate alone doesn't prevent two people booking seat A1 — they both pass the capacity check. The seat lock alone doesn't prevent total overselling — it only guards one specific seat. You need both for different failure modes.

Rollback When Counter Goes Negative

long remaining = await cache.StringDecrementAsync(seatsKey);
if (remaining < 0)
{
    await cache.StringIncrementAsync(seatsKey);  // restore counter
    await cache.KeyDeleteAsync(lockKey);          // release seat lock
    return Results.Conflict("Sold out");
}

This is optimistic concurrency with rollback — let the operation happen, check if valid, undo if not. The alternative (pessimistic — check first, then act) requires two separate operations with a race condition between them.

Full interview answer for distributed counter: "I use Redis with two separate guards. First, an atomic DECR on the event capacity counter — Redis is single-threaded so no two DECRs interleave, and if the result goes negative I immediately INCR to restore and reject the request. Second, a SETNX seat lock per specific seat with a 5-minute TTL — this prevents two users booking the exact same seat. If either guard fails I roll back both. SQL is the source of truth for the booking record — Redis is the fast concurrency gate in front of it. The CAP trade-off is CP — if Redis is down we reject requests rather than risk overselling, because an oversold seat is unrecoverable."

7. Domain-Driven Design (DDD)

DDD is a way of writing code that mirrors how the business thinks. The core problem it solves: business rules scattered across the codebase, duplicated, easy to bypass, impossible to test in isolation.

The Core Problem — Anemic Domain Model

An anemic domain model is a class that has data but no behaviour. "Anemic" means weak, lacking vitality. The object just sits there passively while everything external acts on it.

// Anemic — anyone, anywhere can do this
booking.Status = "Confirmed";
booking.Status = "Banana";      // no rules, compiles fine
booking.Status = null;          // crashes later

The business rule "only confirm a Pending booking" exists as an if-statement somewhere in Program.cs. Every developer who touches status must know to check that file. When someone forgets, bugs appear.

The Solution — Aggregate Root

An object that protects its own state. Nobody changes it directly. They call methods.

public class Booking
{
    public string Status { get; private set; } = "Pending";  // private set

    public void Confirm()
    {
        if (Status == "Confirmed") return;           // idempotent
        if (Status != "Pending")
            throw new InvalidOperationException(...); // rule enforced
        Status = "Confirmed";
    }

    public void Expire()
    {
        if (Status != "Pending") return;  // silent no-op — safe to call twice
        Status = "Expired";
    }
}

private set means nothing outside the class can assign to Status. Confirm() is the only door in. The compiler enforces this — trying to do booking.Status = "Confirmed" directly is a compile error on every single build. A code review is a human reading code — a human can miss a line. The compiler cannot. You physically cannot ship the mistake.

The Core Concepts

Aggregate

A cluster of related objects that you treat as one single unit — loaded together, saved together, and kept correct together. One object in the cluster is the aggregate root (below); the rest are its children.

Plain English: An Order and its OrderLines form one aggregate. You never deal with a lone order line — you always work with the whole order.

An aggregate is defined by two boundaries that turn out to be the same boundary:

  • Consistency boundary — the group of things that must always stay correct with each other. Example: an order's total must equal the sum of its lines. That rule can only be guaranteed if the order and its lines are treated as one unit.
  • Transaction boundary — because those things must stay correct together, they must also be saved together, in one all-or-nothing database transaction. Save half of them and the rule breaks.

So the full definition is simply: an aggregate is the smallest group of objects that must be correct together (the consistency boundary), and therefore must be saved together (the transaction boundary). The two boundaries line up exactly — that is the idea.

Two rules that fall straight out of this:

  • One repository per aggregate root — you load and save whole aggregates, so there's an OrderRepository but never an OrderLineRepository. (Full explanation in §14.)
  • Keep aggregates small — one aggregate = one transaction, so a huge aggregate makes the database lock a huge amount of data on every change. If an aggregate grows to thousands of rows, the boundary is usually drawn too wide: split the children into their own aggregates and link them by ID.

Where the terms come from: Eric Evans, Domain-Driven Design (2003); sharpened by Vaughn Vernon's Effective Aggregate Design. Your Booking is the simplest possible aggregate — it owns no children, so the whole aggregate is just the one object.

One rule, viewed four ways. Everything about the Order/Lines aggregate traces back to a single rule: "the total must equal the sum of the lines." If you save the order and its lines separately, that rule can break. So:
  • Consistency boundary — Order + Lines are the things that must stay correct together, because of that one rule.
  • Transaction boundary — because they must stay correct together, they're saved together, all-or-nothing.
  • No OrderLineRepository — you never work with a line on its own; it only ever exists as part of the order.
  • Private list — forces every change through a method that can keep the rule true (e.g. AddLine() updates Total in the same step).
Four things that sound like separate rules to memorise — but they're really one idea, seen from four angles.

Why bother — isn't this just DDD ceremony?

Fair question. The honest answer: an aggregate is a fix for a specific bug pattern, not a box to tick because "DDD says so." Without one, this compiles and ships:

order.Total = 500;
order.Lines.Add(new OrderLine { Price = 50 });
// Total still says 500. Nothing enforced it. Silent data corruption — the kind
// that doesn't crash, just quietly gives a customer a wrong invoice.

Nothing stops this because the rule "Total = sum of Lines" doesn't live anywhere — every developer has to remember it, everywhere, forever. An aggregate makes it structurally impossible to forget, by only allowing the change through one method that keeps both in sync:

public void AddLine(OrderLine line)
{
    _lines.Add(line);
    Total += line.Price;   // can't be missed — it's the only place this is allowed to happen
}

So it's a real trade-off, not a default. Use an aggregate when:

  • There's a rule that spans more than one field or object (Total ↔ Lines; can't confirm an expired booking) — and it must never be violated.
  • Multiple developers or services will touch this data over time, so you can't rely on everyone remembering the rule.
  • A violation is expensive — corrupted financial data, double-booked seats — not just cosmetic.

Skip it when you're doing plain CRUD with no cross-field rule — a Tag with just a Name, say. Wrapping that in an aggregate is pure ceremony for zero benefit; a plain property bag over direct EF Core (rungs 1–3 of §12) is the right call.

What if the aggregate is genuinely huge?

Say an Order ends up with 200,000 lines. "Keep aggregates small" doesn't mean hoping your data stays small — it's a signal to check whether the boundary is drawn correctly. Ask: do all 200,000 lines really need to change together, in the same instant? Almost always, no — adding line #200,001 doesn't require locking or loading the other 200,000.

The usual fix: make the children their own small aggregates, referenced by ID instead of held in a list:

public class OrderLine   // now its own aggregate — not a child inside Order
{
    public Guid OrderId { get; private set; }   // reference by ID, not by holding the object
    public decimal Price { get; private set; }
}

Adding a line becomes a tiny transaction — one new row — never loading the other 200k. The rule "Total = sum of Lines" then has two honest options instead of loading everything:

  • Targeted update — bump the total in the same statement that inserts the line, touching 2 rows instead of 200,000: 
    INSERT INTO OrderLines (OrderId, Price) VALUES (@id, @price);
UPDATE Orders SET Total = Total + @price WHERE Id = @id;
  • Eventual consistency — the total updates a moment later via a background handler; the UI may briefly show "calculating…". Normal for huge datasets (see §2 on BASE).

Real systems do exactly this: a bank account with 10 million transactions doesn't load all 10M to post one — the balance is a stored number, updated in a targeted write, and each transaction is its own independent record. The takeaway: a genuinely huge, must-be-instantly-atomic aggregate is extremely rare. A huge row count almost always means the boundary was drawn too wide — split the children into their own aggregates and let the roll-up value catch up.

Aggregate Root

The one object in an aggregate that outside code is allowed to touch — the single entry point for changes, with no side door. Your Booking class is one.

Plain English: You can't reach SeatId or Status and change them directly — you have to go through a method on Booking. Because every change goes through the same door, all the rules stay intact. There's no way around them. (In the Order example, Order is the root; you go through it to touch a line, never straight to the line.)

Entity

An object with an ID. Two entities with the same values but different IDs are different things.

Plain English: Your Booking has a Guid Id. Two bookings for the same seat at the same event are still different bookings because they have different IDs.

Test: If two objects have identical values, are they the same? For an Entity — no, because the ID is different.

Value Object

An object with no ID. Defined purely by its values. Immutable — never modified, only replaced.

Plain English: SeatId("A1") equals another SeatId("A1") because the value is the same. Like a £10 note — you don't care which specific note it is, just that it's £10.

Three benefits:

  • Validation at creation — SeatId("") throws, invalid state is impossible
  • Type safety — compiler catches passing an EventId where SeatId is expected
  • Behaviour on the value — seatId.Row, seatId.IsVIP

Test: If two objects have identical values, are they the same? For a Value Object — yes.

Domain Event

Something that happened. Always past tense. BookingConfirmed. BookingExpired.

Plain English: When Confirm() is called, the booking records that something happened. Other parts of the system react — publish to RabbitMQ, send email, update read model. The booking doesn't know or care what happens next. It just records the fact.

This decouples the domain from infrastructure. The booking's job ends when it raises the event.

Bounded Context

A boundary around a part of the system where words have a specific meaning.

Plain English: "Booking" in Booking.API means a reservation with payment, status, and a 5-minute hold window. "Booking" in Catalog.API means "a seat was sold, decrement the count." Same word, different model, different service.

Why your services are separate: They're in different bounded contexts. Each context owns its model completely. Sharing a database would mean sharing the model — breaking the boundary.

Why BookingEvent DTO is duplicated: Each service owns its own copy of the wire contract. Sharing a type library couples two independently deployable services.

Repository Pattern

Hides database implementation behind an interface. The domain knows nothing about SQL, EF Core, or MongoDB.

public interface IBookingRepository
{
    Task<Booking> GetByIdAsync(Guid id);
    Task SaveAsync(Booking booking);
}
// Domain depends on this interface
// EF Core implementation is hidden behind it

Plain English: The Booking aggregate doesn't call DbContext directly. A repository does the saving. Swap SQL Server for Postgres — change one class. Nothing in the domain changes.

Static Factory Method — Why not just a constructor?

// Static factory method
public static Booking Create(string eventId, string seatId, string userId) { ... }

// Usage
var booking = Booking.Create("evt-coldplay", "A1", "user-123");
  • It has a name — Booking.Create() communicates intent. new Booking() does not.
  • Forces valid state — the only door in. All validation happens here.
  • Private constructor — prevents bypassing Create() via new Booking(). EF Core can still call it via reflection — reflection bypasses access modifiers entirely (private, protected, it doesn't matter). EF Core is the framework, not application code, so it's allowed to do things your own code can't.

Why Confirm() is Idempotent but Expire() Never Throws

The question isn't about the method — it's about who's calling and what stale state means for that caller.

MethodCallerBehaviour on stale stateReason
Confirm() Payment webhook Idempotent (silent) if already Confirmed. Throws if Expired. Webhook may fire more than once — silent return on a duplicate call is the right idempotent behaviour.

But if payment arrives for an Expired booking, something genuinely went wrong — a user paid for a seat they no longer hold. That needs to surface as an error, not be silently swallowed.
Expire() Background sweep worker — runs every 30s across thousands of rows Silent return always — never throws If the worker hits a row that's already Expired or Confirmed, that just means the previous sweep already handled it. Throwing would crash the worker over nothing.

Same situation (stale state), different caller, different right answer.

Interview answer — "Walk me through your Booking class": "Booking is an aggregate root. All properties have private setters — nothing outside the class can change state directly. Business rules live on the object through methods: Confirm() enforces that only a Pending booking can be confirmed and is idempotent so calling it twice is safe — important for payment webhooks that may fire more than once. Expire() is a silent no-op if already expired — the background worker processes many bookings and should never crash on a stale row. If any developer tries to set Status directly, it's a compile error. The rules are enforced by the compiler, not by convention or code review."

8. Clean Architecture

Clean Architecture is a way of structuring a codebase so that business rules never depend on infrastructure. The core rule: inner layers know nothing about outer layers. Dependencies only point inward.

The Problem It Solves

Before the split, everything lived in one project: Booking.cs, BookingDbContext, Program.cs endpoints, and workers all in Booking.API. The problems:

  • Business logic and EF Core calls were in the same files — changing the database means touching the same file as your business rules
  • Workers referenced BookingDbContext directly — persistence and domain logic were tangled
  • No seam to test business rules without a real database

The Three-Layer Split

┌──────────────────────────────────────────────────────┐
│                    Booking.API                       │
│   HTTP endpoints · workers · DI wiring (Program.cs)  │
│   References: Domain + Infrastructure                │
├──────────────────────────────────────────────────────┤
│               Booking.Infrastructure                 │
│   BookingDbContext · BookingRepository · Migrations  │
│   References: Domain only                            │
├──────────────────────────────────────────────────────┤
│                  Booking.Domain                      │
│     Booking aggregate root · IBookingRepository      │
│     Zero NuGet dependencies                          │
└──────────────────────────────────────────────────────┘

Dependency direction:
  API  ──▶  Infrastructure  ──▶  Domain
  API  ──▶  Domain

Domain knows nothing. Infrastructure knows Domain. API knows both.

What Moved Where

Why does IBookingRepository live in Domain, not Infrastructure? The domain defines what it needs — a way to save and retrieve bookings. It doesn't define how. The interface is the contract. If it lived in Infrastructure, Domain would have to reference Infrastructure to use it — that reverses the dependency direction. The rule is: Infrastructure depends on Domain, never the reverse.

AddBookingInfrastructure — Extension Method

// Booking.Infrastructure/InfrastructureExtensions.cs
public static IServiceCollection AddBookingInfrastructure(
    this IServiceCollection services, string connectionString)
{
    services.AddDbContext<BookingDbContext>(options =>
        options.UseSqlServer(connectionString));
    services.AddScoped<IBookingRepository, BookingRepository>();
    return services;
}

// Program.cs — one line, no EF Core knowledge here
builder.Services.AddBookingInfrastructure(
    builder.Configuration.GetConnectionString("SqlServer")!);

Program.cs has zero knowledge of BookingDbContext, UseSqlServer, or BookingRepository. All infrastructure wiring is encapsulated in one method in the Infrastructure project. Switching to Postgres means changing UseSqlServer to UseNpgsql in one file — nothing in Program.cs or Domain changes.

Workers are Singleton — they cannot inject Scoped services directly. IBookingRepository and BookingDbContext are Scoped (one instance per request). A Singleton lives forever — holding a Scoped service inside it means that service is never disposed, leaking DB connections. ASP.NET Core throws a captive dependency exception at startup if you try. Solution: inject IServiceScopeFactory (which is Singleton-safe), then create a new scope at the start of each worker cycle. The scope — and the DbContext inside it — is disposed at the end of the using block. 
// Workers do this each cycle:
using var scope = _scopeFactory.CreateScope();
var repository = scope.ServiceProvider.GetRequiredService<IBookingRepository>();
// scope.Dispose() at end of using block → DbContext is cleaned up
Interview answer — "Why Clean Architecture?": "I split Booking.API into three projects: Domain, Infrastructure, and API. Domain has zero NuGet dependencies — pure C# with the Booking aggregate and the IBookingRepository interface. Infrastructure implements that interface using EF Core. API wires everything together with one call to AddBookingInfrastructure(). The dependency direction is always inward: API depends on Infrastructure and Domain, Infrastructure depends on Domain, Domain depends on nothing. If we swap SQL Server for Postgres, nothing in Domain or the HTTP handlers changes — only one method in Infrastructure."

9. Repository Pattern

Why it's called a repository

A repository in the physical world is a place you go to fetch and store things — a library, a warehouse, a filing cabinet. You don't care how things are organised inside. You say "give me the booking with ID 123" and it comes back. You say "store this booking" and it disappears inside. What happens in there is not your problem.

That's exactly what IBookingRepository does. The endpoint says "give me a booking" — it doesn't know or care whether that came from SQL Server, MongoDB, an in-memory list, or a text file. The repository hides all of that.

// The endpoint thinks it's talking to "a place that holds bookings"
var booking = await repository.GetByIdAsync(id);

// It has no idea this is running underneath:
return await _context.Bookings.FindAsync(id);  // SQL Server, EF Core, connection pools — all hidden

The interface is described as collection-like because calling repository.AddAsync(booking) feels like adding to a list — myList.Add(booking). The caller thinks in terms of "a collection of bookings", not "a database table with rows and SQL". The repository creates that illusion: a simple get/add/remove surface, regardless of what's actually inside.

Crucially, the repository holds domain objects — not raw database rows, not DTOs. What comes back from GetByIdAsync() is a Booking domain object, not a plain bag of column values. The distinction matters because a raw row from a DataReader is just data: Id, Status, CreatedAt as primitive values with no behaviour attached. A domain object carries both state and rules. But to be precise — Confirm() and Expire() are not part of the repository. The repository's job is storage, retrieval, and search. You fetch the object from the repository, then call domain behaviour on it, then hand it back to the repository to persist. Three separate steps.

var booking = await repository.GetByIdAsync(id);  // 1. repository retrieves
booking.Confirm();                                 // 2. domain behaviour — nothing to do with repository
await repository.SaveAsync();                      // 3. repository persists the change

Martin Fowler named and defined the pattern in 2002: "a mechanism for encapsulating storage, retrieval, and search behaviour, which emulates a collection of objects." Every developer since has used the same word for the same idea — so when you see IBookingRepository you immediately know what it is without reading the code. The name is convention. The actual value is the interface that lets everything else pretend the database is just a simple collection of domain objects.

The Interface (in Domain)

public interface IBookingRepository
{
    Task AddAsync(Booking booking);
    Task SaveAsync();
    Task<Booking?> GetByIdAsync(Guid id);
    Task<List<Booking>> GetPendingBookingsOlderThan(DateTime cutoff);
    Task<List<Booking>> GetConfirmedUnpublishedBookings();

The domain defines what queries it needs. The implementation — EF Core, SQL queries — lives in Infrastructure. Domain never references EF Core.

The explicit BeginTransactionAsync / CommitAsync was removed.     booking.Expire();

The raw LINQ query (db.Bookings.Where(...)) moved into BookingRepository.GetPendingBookingsOlderThan(cutoff). The worker has no idea how bookings are fetched. The single save after the loop is intentional — calling SaveAsync() inside the loop would make N database roundtrips instead of one. EF Core tracks all the Expire() changes in memory; one SaveChangesAsync() flushes them all.

    await db.SaveChangesAsync();
}
// AFTER

    // publish to RabbitMQ ...

}

This worker deliberately saves inside the loop, unlike the ExpiryWorker. Reason: if the process crashes mid-loop, already-published bookings have their PublishedAt stamped — they won't be re-published on the next cycle. Crash safety outweighs batching here.

SaveAsync() — Why No Parameter?

// Interface
Task SaveAsync();

// Implementation
public async Task SaveAsync() =>
    await _context.SaveChangesAsync();

SaveChangesAsync() doesn't save a specific booking — it saves all changes tracked by the DbContext. Passing a Booking parameter would imply only that booking is saved, which is misleading. No parameter makes the API honest: "flush everything you're tracking." The caller's pattern is always: call a domain method to change state, then call SaveAsync() to persist those changes.

The "never use repository with EF Core" argument

The short version: "There is literally no point. Just use the DbContext directly. You'll be fine." DbContext is already a repository and Unit of Work. Adding an interface on top is an abstraction over an abstraction.

Here is what the two approaches look like side by side:

10. Unit of Work — and the Nick Chapsas Debate

What is the Unit of Work pattern?

Unit of Work is a pattern that answers one question: when multiple things change in a single business operation, how do you ensure they are all saved together or not at all?

The pattern works by tracking every object that was changed during an operation — created, modified, or deleted — and then flushing all of them to the database in one coordinated transaction when you say "commit". If anything fails, everything rolls back. No partial saves.

Without it, you'd call the database once per change — booking created (1 DB call), status updated (2nd DB call), outbox event written (3rd DB call). If the process crashes between calls 2 and 3, the database is in an inconsistent state. Unit of Work wraps all three into one atomic operation.

How DbContext IS the Unit of Work

EF Core's DbContext implements the Unit of Work pattern natively. You don't need to build one. It works in three phases:

PhaseWhat happensIn code
1. Track Every entity you load or add is registered in the change tracker — an in-memory dictionary of every object the DbContext knows about, along with its original vs current values db.Bookings.Add(booking) or loading via FindAsync()
2. Mutate You change properties on tracked entities in C#. EF Core detects the difference between the original snapshot and the new values automatically — you don't call any "mark as dirty" method booking.Confirm() sets Status = "Confirmed" — EF Core notices
3. Flush One call generates and executes the SQL for every pending change in a single database transaction. All succeed or all roll back await db.SaveChangesAsync() 
// The Unit of Work in action — three changes, one SaveChangesAsync
var booking = await db.Bookings.FindAsync(id);   // tracked by EF Core
booking.Confirm();                               // change tracked (no DB call yet)

var outbox = new OutboxEvent(booking.Id, ...);
db.OutboxEvents.Add(outbox);                     // added to tracker (no DB call yet)

await db.SaveChangesAsync();
// ↑ EF Core now generates two SQL statements inside ONE transaction:
//   UPDATE Bookings SET Status='Confirmed' WHERE Id=...
//   INSERT INTO OutboxEvents (...)
// If either fails, both are rolled back.

This is exactly what repository.SaveAsync() calls under the hood in this project — it delegates to db.SaveChangesAsync(), which is the Unit of Work flush.

The Nick Chapsas debate — full breakdown

Nick Chapsas (a prominent .NET YouTuber) argues that wrapping EF Core in a repository pattern is adding an abstraction over something that is already a well-designed abstraction. His specific points:

ArgumentWhat it means
"DbContext is already a repository" db.Bookings is a DbSet<Booking> — it already provides Add, FindAsync, Where, Remove. IBookingRepository wraps these with methods that do the same thing. You've written an abstraction on top of an abstraction for no new capability.
"DbContext is already a Unit of Work" As shown above — the change tracker + SaveChangesAsync IS the Unit of Work. A separate IUnitOfWork interface wrapping it adds boilerplate without adding behaviour.
"You're never actually swapping the database" The classic justification for the repository is "so you can swap EF Core for Dapper later." In practice this almost never happens — and when it does, the repository interface leaks so many EF Core assumptions (tracked entities, async patterns, IQueryable shapes) that it doesn't actually shield you.
"You lose EF Core's best features" Behind a GetByIdAsync(Guid id) method, callers can't request projections (.Select(b => new Dto(...))), skip tracking (.AsNoTracking()), split queries, or write complex LINQ. You'd have to add a new method to the interface every time a new query shape is needed — the interface grows forever.
"Unit testing argument is weaker than it looks" The repository is often justified as "it lets us mock the database in tests." But EF Core ships an in-memory provider and a SQLite in-memory mode — you can test against a real (lightweight) database without mocking. Mocking the repository hides bugs where your LINQ would fail against a real database.

The counter-arguments — why teams still use it

ReasonWhen it holds
Architectural boundary enforcement In Clean Architecture, Domain must not reference EF Core. IBookingRepository in Domain is the seam — it defines what the domain needs without naming how. Without it, workers and handlers would import Microsoft.EntityFrameworkCore directly, which breaks the dependency rule.
Multiple database backends If the same interface needs SQL and MongoDB implementations (e.g. read from Mongo materialized view, write to SQL), a repository interface is the natural place to express that. DbContext only works with relational databases.
Team consistency On large teams, named methods like GetPendingBookingsOlderThan(cutoff) are more discoverable and harder to misuse than raw LINQ scattered across handlers. The repository becomes a library of vetted query patterns.
Testability via interface For unit testing domain logic in isolation — where you want zero database involvement — an interface you can mock or stub is simpler than configuring an in-memory EF Core provider. This project's BookingExpiryWorker logic could be unit-tested with a fake IBookingRepository that returns hardcoded data.

Why Catalog.API's repository doesn't add a Unit of Work

Catalog.API has a repository (ICatalogRepository) but no separate IUnitOfWork interface. The reason: DbContext is already the Unit of Work. Wrapping it in another interface would mean writing this:

// Pointless wrapper — all it does is delegate to DbContext
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync();
}

public class UnitOfWork : IUnitOfWork
{
    private readonly CatalogDbContext _context;
    public UnitOfWork(CatalogDbContext context) => _context = context;

    public Task<int> SaveChangesAsync() => _context.SaveChangesAsync();
}

This is pure boilerplate. DbContext.SaveChangesAsync() already coordinates all tracked changes as a Unit of Work. Adding a wrapper class that calls it achieves nothing. Catalog.API instead calls _context.SaveChangesAsync() directly inside the repository implementation — the Unit of Work is being used, just without a redundant interface on top of it.

Booking.API's IBookingRepository.SaveAsync() is the same idea expressed slightly differently — it hides the EF Core call behind the interface, but for an architectural reason (Domain cannot reference EF Core), not to introduce a separate Unit of Work pattern.

Interview answer — "What is the Unit of Work pattern and does EF Core implement it?"
"Unit of Work tracks all changes made during a business operation and flushes them to the database in one transaction. EF Core's DbContext implements it natively — the change tracker records every mutation, and SaveChangesAsync commits them all atomically. I don't add a separate IUnitOfWork wrapper on top because DbContext already is one. Adding a wrapper just creates boilerplate that delegates straight through. Where I do use a repository interface — like IBookingRepository in Booking.API — it's to enforce the Clean Architecture boundary: Domain must not reference EF Core, so the interface lives in Domain and the EF Core implementation lives in Infrastructure."
Interview answer — "Do you agree with Nick Chapsas that you should never use a repository with EF Core?"
"Mostly yes, in simple cases. DbContext is already a repository and unit of work — adding IRepository on top is wrapping an abstraction with another abstraction, and you lose EF Core features like projections, AsNoTracking, and IQueryable composability. The 'swap the database' argument rarely holds up in practice. But I'd use the repository interface when there's a genuine architectural reason — like Clean Architecture's dependency rule, where Domain cannot reference EF Core directly, or when the same interface needs multiple implementations across different storage backends. The key is that the interface earns its existence through a real constraint, not just as a habit from pre-EF patterns."

11. Vertical Slice Architecture

The core idea

Vertical Slice Architecture organises code by feature — what the code does — rather than by technical role — what kind of thing it is. Each feature gets its own folder containing everything it needs.

The name comes from the image of slicing through layers. A horizontal layer (Controllers, Repositories, Models) spans the whole app. A vertical slice cuts through all layers for just one feature.

Horizontal layers — the default

The traditional approach groups files by technical role:

Catalog.API/               ← BEFORE (horizontal)
  CatalogItem.cs           ← all models together
  CatalogSyncWorker.cs     ← all workers together
  Program.cs               ← endpoint + seeding + wiring mixed together
  Repositories/
    ICatalogRepository.cs  ← all repositories together
    CatalogRepository.cs

The problem: when you want to understand or change the "sync booking" feature, you read CatalogSyncWorker.cs, then jump to Repositories/ for the data access, and then back to Program.cs for the endpoint. Code for one feature is scattered across multiple folders. Adding a new feature means touching multiple folders.

The folder structure tells you what kinds of things exist, not what the system actually does.

Vertical slices — organised by feature

    SyncBooking/

The benefit: to understand or change "sync booking", you open Features/SyncBooking/ and everything is there. Adding a new feature means adding one new folder and touching nothing else. The folder structure tells you exactly what the system does at a glance.

The test: Can a new developer open the project and understand what it does just from the folder structure? With horizontal layers: no — they see "Repositories", "Models", "Workers" which describes the architecture, not the business. With vertical slices: yes — they see "GetItems", "SyncBooking", "Seed" which describes the features.

What goes in Shared/?

Shared/ holds things used by more than one feature. The rule is strict: if only one feature uses something, it lives inside that feature's folder. If multiple features use it, it lives in Shared/.

Static classes in vertical slices

Feature files like GetItemsEndpoint and CatalogSeeder are static classes. This is intentional. A static class has no state — no instance fields — and cannot be instantiated with new. It's a container for utility methods, not an object with behaviour.

State means instance fields — data stored on an object between method calls. GetItemsEndpoint holds nothing. It has one job: register a route at startup. Once done, it's never used again. Static makes that explicit.

Contrast with CatalogRepository, which holds _database and _logger as instance fields. Those are state. The repository must be an instance class so the DI container can inject dependencies into it and so it can implement the ICatalogRepository interface (static classes cannot implement interfaces).

12. The Repository Ladder — 9 Patterns in One Debate

A famous r/dotnet argument — "Why is the Generic Repository still the default in .NET tutorials?" — looks like people fighting about one thing. It's really nine different patterns stacked in a ladder, and the people arguing are each standing on a different rung. Each rung fixes a problem the rung below it caused. Once you see the ladder, the whole argument makes sense — and you'll know which rung fits your own project.

Don't try to memorise this table — it's just a map of the journey. Each rung is explained in plain language below.

The whole ladder at a glance

#RungThe problem it fixes (created by the rung below it)
1Active RecordStarting point — the object saves itself.
2RepositoryRules and saving are stuck together, so you can't test a rule without a database.
3Generic Repository hatedYou keep copy-pasting the same Save/GetById/GetAll for every object.

5Rich Domain Model / AggregateThe object is wide open — nothing stops bad data being set.

Rung 1 → Active Record

The simplest possible design: the object holds its data and knows how to store itself. State-changing methods persist themselves.

public class Booking
{
    public string Status = "Pending";

    public void Confirm()
    {
        if (Status == "Expired")
            throw new InvalidOperationException("Cannot confirm expired booking.");
        Status = "Confirmed";
        Database.Write(this);   // ← the method saves ITSELF. That's Active Record.
    }
}

The problem: say you just want to test one rule — "confirming a pending booking makes it Confirmed." You can't, because Confirm() also writes to the database at the end. So testing that one small rule forces you to set up a whole database. The rule and the saving are glued together in the same method.

Rung 2 → Repository

Cut the storage out of the object into a separate class. The booking goes back to being pure data + rules; a repository owns persistence.

public class Booking
{
    public string Status = "Pending";   // NO Save(). It doesn't know the DB exists.
}

public class BookingRepository
{
    public void Save(Booking booking) => Database.Write(booking);   // storage lives HERE
}

A "repository" is literally a place things are kept — hand it an object to store, or ask it to fetch one back. Now the booking is testable in isolation because it no longer references the database. The pain it leaves: you have Booking, Event, User, Payment… and you're copy-pasting the same Save/GetById/GetAll for each one.

Rung 3 → Generic Repository the trap

Write it once with a type parameter, reuse for every entity:

public class Repository<T>                       // T = Booking, Event, User…
{
    public void Save(T entity)   => Database.Write(entity);
    public T    GetById(int id)  => Database.Find<T>(id);
    public List<T> GetAll()      => Database.ReadAll<T>();
}

Feels like a win — no more copy-paste. But here's the catch. Because this one class has to work for every type (Booking, Event, User…), it can only have methods that make sense for all of them: Save, GetById, GetAll. So the moment you need something specific — "confirmed bookings for the taylor-swift event" — there's no method for it, and there can't be: a generic Repository<T> has no idea what "Confirmed" or "EventId" even mean. You're left with two bad options:

// BAD MOVE A — load everything, filter in memory (the performance trap)
var all = repo.GetAll();                        // drags ALL rows out of the database
var result = all.Where(b => b.EventId == "taylor-swift" && b.Status == "Confirmed");
// 5 million rows into memory to keep 2. Server melts.

// BAD MOVE B — bolt a LINQ expression on (the leak)
public List<T> Find(Expression<Func<T, bool>> predicate);   // caller now writes the query
// The "abstraction" abstracts nothing — EF leaks straight back through it.

Why GetAll() loads everything — IEnumerable vs IQueryable

Bad Move A said "load everything, filter in memory." But why does it load everything? The answer is hidden in one detail: the return type. That single word decides whether the database does the filtering or your app does.

  • IEnumerable<T> (and List<T>) means the data is already fetched. The query has already run. Any .Where(...) you write after this runs in your app's memory, on rows that are already sitting there.
  • IQueryable<T> means a query that hasn't run yet — it's lazy. Any .Where(...) you add becomes part of the SQL, so the database does the filtering and only the matching rows come back.

GetAll() returns IEnumerable<T>, so it fetches every row first, then filters in memory:

// GetAll returns IEnumerable — the query runs HERE, pulling ALL rows in:
var all = repo.GetAll();                              // SELECT * FROM Bookings → 5,000,000 rows into memory
var result = all.Where(b => b.Status == "Confirmed"); // filter runs in C#, on rows already loaded

// EF directly — db.Bookings is IQueryable, so the filter joins the SQL:
var result = await db.Bookings
    .Where(b => b.Status == "Confirmed")   // becomes part of the query: WHERE Status = 'Confirmed'
    .ToListAsync();                         // runs NOW; the database filters; only matching rows return

Identical-looking code, completely different cost. IEnumerable = fetch everything, then filter. IQueryable = filter first, then fetch only what matches. A repository that hands back IEnumerable throws away EF's lazy querying — the very thing you were paying for.

The leaky-abstraction rot (why generic repos are hated)

Watch what happens to that one simple method as real needs pile on, one at a time:

// Stage 1 — clean and simple
Task<List<T>> GetAllAsync();

// Stage 2 — need related data, so bolt on includes (magic strings — typo = runtime crash)
Task<List<T>> GetAllAsync(params string[] includes);

// Stage 3 — need filtering, so expose a LINQ predicate (caller now writes LINQ)
Task<List<T>> GetAllAsync(Expression<Func<T, bool>> predicate, params string[] includes);

// Stage 4 — need sorting + paging, so bolt those on too
Task<List<T>> GetAllAsync(Expression<Func<T, bool>> predicate,
    Func<IQueryable<T>, IOrderedQueryable<T>> orderBy, int skip, int take, params string[] includes);

By stage 4, your "simpler" method is just a clumsier copy of the tools EF already gave you. You rebuilt filtering, sorting, and paging by hand — and worse than the originals. The wrapper was supposed to hide the database details, but every one of them crept back in through its parameters. When a wrapper leaks the very thing it was meant to hide, that's called a leaky abstraction — and it's the main reason people hate the generic repository.

It also hides EF Core's best features

There's a second cost to wrapping EF in a generic repository: you lose easy access to the powerful things EF can already do. Several of EF's most useful tools only work when you call them directly on the query:

  • .Include(...) — pull in related data (an order and its lines) in one trip.
  • .AsSplitQuery() — split one huge join into a few smaller queries, which is often much faster.
  • .TagWith("...") — stick a label on a query so you can spot it later in the database logs.
  • .IgnoreQueryFilters() — turn off automatic filters (for example, to let an admin see deleted rows).

A plain GetById / GetAll gives you no way to reach any of these. So you either go without them, or you start bolting on more parameters to let them through — which is the same leaky slide from above. Either way you end up worse off than if you'd just used EF directly. This is sometimes called the "least common denominator" problem: to cover every type, the wrapper can only offer the few things they all share, so all the powerful, specific features get left out.

Rung 4 → Specific Repository

Give up on "write it once." Hand-write a repository per aggregate, with methods named for the job and the LINQ sealed inside:

public interface IBookingRepository
{
    Task<List<Booking>> GetConfirmedForEventAsync(string eventId);   // named intent, no leak
}

public class BookingRepository(AppDb db) : IBookingRepository
{
    public Task<List<Booking>> GetConfirmedForEventAsync(string eventId) =>
        db.Bookings
          .Where(b => b.EventId == eventId && b.Status == "Confirmed")   // LINQ hidden INSIDE
          .ToListAsync();
}

The caller just says GetConfirmedForEventAsync("taylor-swift"). The database does the filtering, and none of the query details leak out. The cost: you write one method per kind of query you need. That's the trade-off — and it's the sensible choice, so it's what this codebase actually uses. The anger in that thread is aimed at the generic repo (rung 3), not this one. Two other approaches also keep the details hidden:

  • Filter object — one FindAsync(BookingFilter f) method, where BookingFilter is a simple object holding the fields you want to search on (event id, status, and so on). The caller fills in what to look for; the repository works out how. Handy for search screens with lots of optional filters.
  • Composition callback — you hand the repository a small function that shapes the query. Full flexibility, but the query type still shows through to the caller, so it's a middle-ground rather than a fully clean hide.
The simple rule for leaky vs clean: the caller should only say what it wants (like an event id). The repository decides how to get it (the query, kept inside). The moment the how — a raw query expression — shows up in the method's parameters, you've leaked.

Rung 5 → Rich Domain Model / Aggregate

Rung 4's repo saves and fetches fine, but it does nothing to stop a booking being put into a nonsense state, because the object is still a wide-open property bag:

booking.Status = "Confirmed";   // this booking was EXPIRED. no rule stopped it. corrupted data, saved forever.

An object that's all public get/set with no rules inside is called an Anemic Domain Model — basically a database row pretending to be an object. The rules have nowhere to live. The fix: lock the setters so the only way to change the object is by calling a method that checks the rules first.

public class Booking
{
    public string Status { get; private set; } = "Pending";   // READ from outside, WRITE only inside

    private Booking() { }                                       // can't new-up a bad one
    public static Booking Create(...) => new() { ... };         // the only door in — always starts Pending

    public void Confirm()                                       // the only door to change state
    {
        if (Status == "Expired")
            throw new InvalidOperationException("Cannot confirm an expired booking.");
        Status = "Confirmed";
    }
}

Now booking.Status = "Confirmed" won't compile — the set is private. Bad state is unreachable, enforced by the compiler. An object like this — one that holds its own rules and only lets you change it through guarded methods — is a Rich Domain Model.

Aggregate and Aggregate Root — plain version

Some business objects own other objects. Take an Order that owns its OrderLines:

Order  (total: $150, status: Pending)
  ├─ OrderLine  (2 × T-shirt, $40)
  ├─ OrderLine  (1 × Shoes,   $90)
  └─ OrderLine  (1 × Hat,     $20)

There's a rule tying them together: the order's total must always equal the sum of its lines. So who is allowed to add a line?

order.Lines.Add(newLine);   // ❌ BAD — line added, but total NOT updated. Now they disagree → corrupt.
order.AddLine(newLine);     // ✅ GOOD — the method adds the line AND updates the total, together.

The Order + its OrderLines, treated as one protected unit, is an Aggregate. The Order — the only object outside code is allowed to touch — is the Aggregate Root. You never modify a line directly; you go through the root, and the root keeps everything consistent. Your Booking is the simplest possible aggregate — it owns no child objects, so the root is the whole thing. See §14 for why this "one consistent unit" is exactly what a repository should load and save.

The ceasefire line. Even the person who started the thread — after 200 comments bashing repositories — admitted: "DDD Aggregates are basically the only valid use case for a Repository in 2025." Here's why the aggregate and the repository need each other:
  • The aggregate (rung 5) stops you creating bad data inside your running program — booking.Status = "..." won't even compile.
  • The repository (rung 4) is the only code allowed to load and save that aggregate.
Put the two together and there is exactly one path to change data — and that path is forced to go through the rules. 
// The ONLY way to change a booking — every step is guarded:
var booking = await repo.GetByIdAsync(id);   // 1. the repository loads it  (the only door IN)
booking.Confirm();                            // 2. change it through a method that checks the rules
await repo.SaveAsync();                        // 3. the repository saves it  (the only door OUT)

// There is no other route in or out:
booking.Status = "Confirmed";                 // ❌ won't compile — the setter is private
db.Bookings.Update(somethingHandBuilt);       // ❌ not allowed — nothing else talks to the DB directly

That single, rule-enforced path is what people mean when they call the repository the aggregate's guarded door: to touch the data at all, you must go through the object that refuses to hold bad data.

Rung 6 → CQRS

Every booking is now a guarded aggregate — great for changing data, heavy for just showing it. Loading a full aggregate (private constructors, change-tracking, rules) is overkill to render a dashboard list. So split the two paths:

// WRITE path — through the aggregate + repository. Rules enforced.
var booking = await repo.GetByIdAsync(id);
booking.Confirm();
await repo.SaveAsync();

// READ path — skip all of it. Thin, direct, read-only.
var list = await db.Bookings
    .AsNoTracking()                                    // don't track — we're only displaying
    .Where(b => b.EventId == "taylor-swift")
    .Select(b => new BookingListItem(b.Id, b.Status))  // straight into a flat DTO, no aggregate
    .ToListAsync();

CQRS stands for Command Query Responsibility Segregation — a big name for a simple idea: use two different paths for changing data versus reading it. Changing data (commands) goes through the aggregate, because the rules matter. Reading data (queries) goes straight to the database, because speed matters and there are no rules to protect when you're only looking. This pays off at scale: a ticketing site might do 1000 reads for every 1 write, and once the two paths are separate you can speed each one up on its own. (More on CQRS on Day 6.)

Rung 7 → Clean Architecture

By now you have several pieces: the aggregate, the repository interface, the EF Core code that implements it, and the read models. Two questions come up: where does each piece live, and what is allowed to depend on what? Clean Architecture answers both with one simple rule:

Your business rules (the Domain) depend on nothing. Everything else depends on them — never the other way around.

So the Booking aggregate is plain C# that knows nothing about EF Core, MongoDB, or the database. The database code depends on the domain, not the reverse. That keeps your most important code (the rules) clean and easy to test. Full detail is in §8; the interface trick that makes the "depend on nothing" rule possible is in §13.

Rung 8 → The Write-Only Repository (reads go elsewhere)

Rung 6 (CQRS) said "split reads from writes." Rung 8 is the concrete, everyday shape of that split — and it's what experienced DDD developers actually build. It comes down to two small interfaces with one job each:

// WRITE side — the repository. Its ONLY job: hand you an aggregate to change, and save it.
public interface IBookingRepository
{
    Task<Booking?> GetByIdAsync(Guid id);   // load one aggregate, so you can change it
    void Add(Booking booking);
    Task SaveAsync();
    // ← notice: no GetAll, no Find, no queries. Reading is NOT its job.
}

// READ side — a tiny separate query interface. Not the repository.
public interface IQuery<T>
{
    IQueryable<T> Query();   // just returns db.Set<T>().AsNoTracking()
}

// Reading a list goes through the query side — never the repository:
var events = await query.Query()
    .Where(b => b.Status == "Confirmed")
    .Select(b => new EventRow(b.Id, b.Name))   // straight into a flat display object
    .ToListAsync();

Why split it this way? The repository stays tiny and focused: it only does the three write operations that are the same for every aggregate — load-by-id, add, save. There's nothing to leak, because there are no queries in it. All the varied, messy querying lives on the read side, where it can't creep into your write model. This is the exact setup Microsoft's own DDD guide recommends, and it's what one thread commenter meant by "I don't use the repository to read data — I always use a query interface built on top of EF."

This also settles the "but I like a generic repository for DDD" confusion. Some people say they use a generic repository and it works great — which sounds like it contradicts rung 3. It doesn't. Their "generic" base only covers those three write operations (and only for aggregate roots) — GetById, Add, Save. Those are genuinely identical for every aggregate, so sharing them leaks nothing. The hated generic repo (rung 3) is a different beast: it tries to be your query API too (GetAll, Find(Expression)) — and querying is exactly where all the leaking happens. Same word, opposite outcome: generic writes = fine, generic queries = trouble.

Rung 9 → Reusing Reads Without a Repository (Extension Methods)

Rung 8 said: for reading, skip the repository and query the database directly. Great — but what happens when the same complicated query is needed in ten different places? For example: "active gold members who ordered in the last 30 days." Do you copy-paste that everywhere? And if you're not using a repository, where does shared query logic even live?

The answer: write it once as an extension method — a small helper you can tack onto a query. You give the query a friendly name, reuse it anywhere, and it still runs in the database.

// Write the reusable filter ONCE, with a clear name:
public static class UserQueries
{
    public static IQueryable<User> ActiveGoldMembers(this IQueryable<User> users) =>
        users.Where(u => u.IsActive && u.Tier == Tier.Gold);
}

// Reuse it anywhere — reads like plain English, and still runs as ONE SQL query:
var users = await db.Users
    .ActiveGoldMembers()             // ← the shared filter, reused
    .Where(u => u.SignedUpThisYear)  // ← plus anything extra you need this time
    .ToListAsync();

Why this is nice:

  • No repository — the shared logic lives in a small, plain helper, not a wrapper class.
  • Still fast — the query stays lazy (see the IQueryable note in rung 6), so the whole thing runs as one SQL query in the database. Nothing gets loaded into memory just to be filtered.
  • Stackable — you can chain these filters together like Lego blocks.

This trick has a fancy name — the Specification Pattern — but it's really just "reusable, named pieces of a query." It's the read-side answer that lets you drop the repository for reads without copy-pasting the same query everywhere.

Where this comes from. The person who started that r/dotnet thread later wrote an article, "Stop Using the Repository Pattern with Entity Framework Core." Despite the dramatic title, his actual advice matches this ladder almost exactly: for reads, skip the repository and query EF directly, using extension methods like the one above to avoid repeating yourself; for writes, he still uses an aggregate. So "stop using repositories" really means "stop using a generic repository as your query tool" — not "never use one at all." The headline is bigger than the argument. His other three points are all covered above: that a wrapper hides EF's features and that GetAll() loads the whole table are both in rung 3; that mocking the database gives false confidence is in the testing sub-ladder just below.

The testing sub-ladder

Running in parallel through the thread is a second argument — how do you test data access? — which is its own four-rung ladder:

RungApproachVerdict
aMock the repository / DbSetFastest, but mocks lie: they run LINQ-to-Objects while production runs LINQ-to-SQL. A query that passes the mock can throw a translation error in prod.
bEF Core In-Memory providerCloser, but Microsoft's own docs say don't — Include behaves differently, no real query translation.
cSQLite in-memoryA better middle rung — a real relational engine, still fast.
dTestcontainers (real DB in a container)The thread's favourite. Spin up real Postgres/SQL Server per run. Catches translation errors mocks miss.

Why do mocks "lie"? A fake database runs plain C#, but a real database runs SQL — and the two don't always agree. Say you write a query that compares text while ignoring upper/lower case. Against the fake it works perfectly (real C# knows how to do that). Against a real SQL database it might crash, because that exact comparison can't be translated into SQL. Your test went green, but production breaks. That's the trap: the fake hands you confidence the real database won't back up.

The sensible approach: test the two layers separately. Your business rules → use a fake repository (fast, run on every change). The repository itself → test it against a real database in a container (slower, run less often). Don't mix the two. One neat trick from the thread: if a method only needs some data, pull the logic out into a separate method that takes that data as a parameter — then you can test it by just passing the data in, with no database or fake needed.

One more nuance — "DbContext is already a repository" isn't always true

A common argument is "EF Core's DbContext is already a repository, so don't wrap it." That's only true when your objects look exactly like your table rows. But a rich aggregate often doesn't: the way it's stored (flat columns) can differ from the way it lives in memory (an object with private lists, computed values, and rules). Here's a concrete mismatch:

// ── How the Order LIVES IN MEMORY (a rich aggregate) ─────────────────
public class Order
{
    private readonly List<OrderLine> _lines = new();          // PRIVATE list — outsiders can't touch it
    public IReadOnlyList<OrderLine> Lines => _lines;          // only a read-only view is exposed
    public decimal Total => _lines.Sum(l => l.Price);         // COMPUTED on the fly — not a stored field

    public void AddLine(OrderLine line) => _lines.Add(line);  // the only way to add a line
}

// ── How it's STORED (flat rows across two tables) ────────────────────
//   Orders table:      | Id | CustomerId | ... |          ← no "Total" column
//   OrderLines table:  | Id | OrderId | Price | ... |      ← the lines live here, separately

The shapes don't match. A raw DbSet<Order> can't just be your repository, because:

  • _lines is private — EF has to be told to read and write it through its hidden backing field, not a public property.
  • Total is computed, not stored — there's no Total column; it's recalculated from the lines.
  • Loading an order means stitching its separate OrderLine rows back into the aggregate's private list.
// The repository is where that translation lives — it rebuilds the whole aggregate:
public async Task<Order?> GetByIdAsync(Guid id) =>
    await db.Orders
        .Include("_lines")        // pull the child rows and pour them back into the private list
        .FirstOrDefaultAsync(o => o.Id == id);

That translating — flat rows in, whole aggregate out — is a job a raw DbSet doesn't do for you. It's one more reason the aggregate + repository pairing earns its place.

Interview answer — "Do you use the repository pattern?"
"Not a generic one — IRepository<T> over EF Core is a leaky abstraction that hides AsSplitQuery, forces you to either load everything into memory or expose Expression<Func<>>, and ends up re-implementing LINQ badly. But I use a specific repository when I'm doing DDD — the repo hydrates aggregates and is the only door to my domain entities, so invariants stay enforced. Reads I keep separate with direct EF and AsNoTracking — that's CQRS. And I don't lean on repos for testability the way people used to: I mock the repo for business-logic unit tests but test the data layer itself with Testcontainers against a real database, because EF In-Memory doesn't catch translation errors. The people arguing about this are usually just standing on different rungs — simple CRUD is fine with direct EF; a complex domain earns the aggregate."
The meta-insight: people argue endlessly about repositories because they are standing on different rungs and think they're on the same one. "Just use EF directly" (rung 2–3) and "the repo protects my aggregate" (rung 5–9) are both correct — about different situations. The Staff-level skill isn't picking a universal side; it's naming which rung a given situation calls for. Simple admin CRUD? Rungs 1–3. Complex domain with real invariants? Climb to 5–9.

Conclusion — the whole ladder in one thought

There is no single "right" answer to "should I use a repository?" — it depends which rung your situation sits on:

  • Simple CRUD, no real rules? Stay low. Use EF Core directly (rungs 1–3). A repository would just be ceremony.
  • A real domain with rules to protect? Climb to the top: a rich aggregate (rung 5) behind a write-only repository (rung 8), reads through a separate query path (rungs 6 and 9), everything organised by Clean Architecture (rung 7).

The people in that thread aren't really disagreeing — they're standing on different rungs. Your job is simply to name the rung the problem in front of you actually needs. That one skill is worth more than memorising any single pattern.

13. Dependency Injection vs Dependency Inversion

Two ideas with near-identical names that people constantly conflate. They are different things at different levels — one is an architectural principle about which way dependencies point; the other is a technique for handing an object what it needs. You can do either without the other. They just happen to work beautifully together.

Dependency Inversion (DIP)Dependency Injection (DI)
What it isA design principle (the "D" in SOLID)A technique / mechanism
Question it answersWhich direction should dependencies point?How does an object get the things it depends on?
LevelArchitecture — how layers relatePlumbing — how one object is constructed
ConcernsInterfaces, abstractions, layer boundariesConstructors, parameters, the DI container
ResultHigh-level policy doesn't depend on low-level detailA class doesn't build its own dependencies

Dependency Inversion (the principle)

The formal statement: "High-level modules should not depend on low-level modules. Both should depend on abstractions." In plain terms: your important code (business rules) should not reach down to your unimportant code (the database). Instead, the important code defines the terms (an interface), and the database bends up to meet them.

Without inversion, the natural dependency arrow points the wrong way:

// WITHOUT inversion — Domain reaches DOWN to Infrastructure
Booking.Domain      ──depends on──▶   Booking.Infrastructure (EF Core)
// Now EF Core leaks into your business rules. Change the DB → change the domain.

With inversion, you put the interface in the Domain and let Infrastructure implement it. The arrow flips:

// WITH inversion — Infrastructure reaches UP to fulfil Domain's interface
Booking.Domain:            defines  IBookingRepository   (the requirement)
Booking.Infrastructure:    implements IBookingRepository  (using EF Core)
Booking.Infrastructure  ──depends on──▶  Booking.Domain
// Domain depends on NOTHING. EF Core can be swapped and Domain never recompiles differently.

Notice: the dependency now points inward (Infrastructure → Domain), even though at runtime the behaviour flows outward (Domain's interface is executed by Infrastructure's code). That reversal — dependency pointing opposite to the call flow — is exactly why it's called inversion.

Dependency Injection (the technique)

A completely separate question: once a class needs something, how does it get it? The bad way is to build it yourself:

// WITHOUT injection — the class constructs its own dependency (tight coupling)
public class BookingService
{
    private readonly BookingRepository _repo = new BookingRepository();  // hard-wired. can't swap. can't fake.
}

The injected way is to receive it — usually through the constructor — and let something else decide what to hand over:

// WITH injection — the dependency is handed in
public class BookingService(IBookingRepository repo)   // "give me something that keeps the booking promises"
{
    private readonly IBookingRepository _repo = repo;
}

// Registration decides WHICH implementation fills the request:
builder.Services.AddScoped<IBookingRepository, BookingRepository>();   // real
// builder.Services.AddScoped<IBookingRepository, FakeBookingRepository>();  // in a test

The forms of injection: constructor injection (stored in a field, the default), parameter injection (minimal-API lambdas — ASP.NET resolves the parameter per request, nothing stored on the class), and the anti-pattern cousin, the service locator (the class reaches into the container itself with GetRequiredService — works, but hides its dependencies instead of declaring them).

How they relate

DI is the most common mechanism by which DIP is achieved, but they are independent:

  • Inversion without injection: you can honour DIP by hand — new BookingRepository() passed manually into a domain that only knows IBookingRepository. No container involved. The principle is satisfied; the technique isn't a container.
  • Injection without inversion: you can inject a concrete class — BookingService(BookingRepository repo). The container hands it over (that's DI), but you're depending on a concrete low-level type, so DIP is not satisfied.
The one-liner: Dependency Inversion is about which way the arrows point (architecture — put the interface in the inner layer). Dependency Injection is about who hands over the object (plumbing — don't new your own dependencies). Inversion is the goal; injection is usually how you get there. The interface is the hinge both turn on — inversion needs it to flip the arrow, injection needs it to make the object swappable.
Interview answer — "What's the difference between DI and dependency inversion?"
"Dependency Inversion is the SOLID principle — high-level modules shouldn't depend on low-level ones; both depend on an abstraction. In my Booking service that means the repository interface lives in the Domain project and the EF Core implementation lives in Infrastructure, so the arrow points inward and Domain never references EF Core. Dependency Injection is the technique that supplies dependencies at runtime — constructor injection plus the built-in container's AddScoped registration. DI is how I usually satisfy DIP, but they're distinct: I could inject a concrete class and violate inversion, or satisfy inversion by hand without a container. Inversion is the architectural goal; injection is the plumbing."

14. Why the Aggregate Is the Ideal Unit for a Repository

A repository by itself doesn't do much — it's just a layer that talks to the database for you. So why bother having one at all? The answer is the aggregate. The aggregate needs a repository, and a repository only truly makes sense when what it stores is an aggregate. They belong together. Here's why.

(Two words you'll see below: an invariant is just a rule that must always be true. Atomic means all-or-nothing — either every change happens, or none of them do.)

One repository per aggregate root — not per table

A common beginner assumption: "every database table needs its own repository." That's wrong. You make one repository per aggregate root — the main object — not one per table. Back to the Order:

Order  (the aggregate root — the main object)
  ├─ OrderLine
  ├─ OrderLine
  └─ OrderLine

You have an OrderRepository. You do not have an OrderLineRepository — even though order lines have their own table in the database. Why not? Because a single order line means nothing on its own. Nobody ever loads or saves one lonely line. You always load the whole Order (with its lines), change it, and save the whole Order back. The repository always deals with the complete object, so it only exists for the main object — the root.

This is what "a repository stores objects, not rows" means. A generic Repository<T> (rung 3) treats every table as a separate thing to get and save on its own — which is exactly why it can't protect anything. A real repository is built around one main object, so it loads and saves that object and everything attached to it in one piece.

The whole aggregate must be saved together

Remember the rule: the order's total must always match the sum of its lines. That rule only stays true if the order and its lines are saved together, in one go. Imagine saving a new line, then the app crashes before it saves the new total — now the total is wrong. Broken data.

Databases have a tool for exactly this: a transaction means "all these changes happen together, or none of them do" (atomic). So the group that must stay correct together — the aggregate — is exactly the group that should be saved in one transaction.

var order = await orderRepository.GetByIdAsync(id);   // load the whole order + its lines
order.AddLine(newLine);                                // add line AND update total, together (rule enforced)
await orderRepository.SaveAsync();                      // save it all at once — one transaction

That's why SaveAsync() takes no parameter. It doesn't save one specific thing — it saves the whole aggregate you're working on, all at once, or nothing at all.

The repository is the only way in and out

In rung 5 we locked the object down so you couldn't put bad data into it in memory (private setters, guarded methods). But there's still a loophole: what if some code skips the object entirely and writes straight to the database table? The repository closes that loophole — it's the only door to storage for the aggregate.

// ❌ writing to the table directly skips the object — and its rules:
db.OrderLines.Add(new OrderLine(...));   // total never updated, no rule checked — broken data in the DB

// ✅ the only allowed path: go through the main object, via its repository:
var order = await orderRepository.GetByIdAsync(id);
order.AddLine(newLine);                  // rule enforced
await orderRepository.SaveAsync();

So the object protects your data in memory, and the repository protects your data in the database. You need both. The object alone can't stop someone writing straight to the table. A repository wrapped around a wide-open object has nothing to protect. Put them together and there's no gap left. That is why people say "aggregates are the only real reason to use a repository" — the aggregate is what turns the repository from a pointless wrapper into a real guarantee.

The three pieces lock together

PieceWhat it guarantees
Aggregate (private setters + methods)You can't put bad data into the object in memory — the compiler and its methods won't let you.
Repository (one per root, the only door)The object can only be loaded and saved as a whole, and only through this one path — no writing to the tables behind its back.
Transaction (one aggregate = one save)Everything in the aggregate is saved together, all-or-nothing.

Take away any one and a gap opens: without the aggregate, code sets fields directly; without the repository-as-only-door, code writes to the tables behind its back; without the one-save-per-aggregate rule, a half-saved order breaks its own total. All three together mean bad data simply can't happen.

Keep aggregates small. Because the whole aggregate is loaded and saved together, a giant one is slow and gets in other users' way. If an Order held 10,000 line items, every tiny change would load and lock all 10,000 at once. So only put things in the same aggregate if they truly must stay correct together. Everything else should be its own separate aggregate, linked by ID rather than owned directly.
Interview answer — "What's the relationship between aggregates and repositories?"
"I make one repository per aggregate root, not one per table. So there's an OrderRepository, but no OrderLineRepository — an order line doesn't exist on its own, it's only ever part of an order. The repository loads and saves the whole order, with its lines, as one unit. That matters because the aggregate is the consistency boundary — which just means the group of things that have to stay correct together. The order's total has to match its lines, and that only stays true if they're saved together. So the aggregate is also the transaction boundary: one aggregate, one all-or-nothing save. And the repository is the only door to storage for that aggregate, so no stray code can go around the order and change the line rows directly. That's the real reason to use a repository, and it's a DDD idea: without an aggregate, a repository is just a pointless wrapper over EF; with one, it's what makes bad data impossible and every change all-or-nothing. I also keep aggregates small, because one aggregate is one transaction — a giant aggregate means the database locks a giant amount of data on every change."
Footnote — how does EF Core actually load a locked-down object? Fair question: if the setters are private and the constructor is private, how does EF fill the object with data from the database? EF cheats (legally) — it uses reflection, a .NET feature that can bypass "private" and write to fields directly. So EF can still create and fill your aggregate, while your code is stuck going through the factory and guarded methods. And EF saves everything in one transaction automatically: change the aggregate through its methods, call SaveChangesAsync() once, and all the changes commit together. So the "one aggregate = one save" rule above happens for free. (This is just the plumbing — the reason it matters is everything above.)
